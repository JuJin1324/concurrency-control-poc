# ADR-003: 왜 시각화를 먼저 하는가?

**날짜:** 2026-01-06
**상태:** Accepted

## Context

### 전통적 개발 방식

**순서:**
1. 구현 (코드 작성)
2. 문서화 (사후)
3. 다이어그램 (선택 사항)

**문제점:**
- ❌ 코드를 먼저 작성하면 잘못된 방향으로 개발할 위험
- ❌ 구현 후 방향이 틀렸다는 것을 깨달으면 다시 작성 (비용 증가)
- ❌ 문서화를 미루다가 결국 하지 않음
- ❌ 다이어그램이 코드와 불일치 (유지보수 안 됨)

### AI 시대의 변화

**AI의 강점:**
- 비용 없이 다이어그램 생성 가능 (Mermaid, PlantUML)
- 코드보다 다이어그램 생성이 더 빠름
- 시각화 자료를 보고 사람이 검토하기 쉬움

**사람의 강점:**
- 다이어그램을 보면 코드보다 이해가 빠름
- 전체 그림을 보고 방향성 판단 가능
- 잘못된 방향을 조기에 발견 가능

**문제:**
- AI와 협업할 때 어떤 순서로 개발해야 효율적인가?
- 잘못된 방향으로 구현하는 것을 어떻게 방지할 수 있는가?
- 인지 부하를 최소화하려면 어떻게 작업을 나눠야 하는가?

## Decision

**AI 시대의 개발 순서: 작은 단위로 시각화 → 검토 → 구현 반복**

### 워크플로우 철학

#### 1. 시각화 우선 (Visualization First)
- AI가 다이어그램을 먼저 생성
- 사람이 다이어그램을 보고 검토
- 코드보다 다이어그램이 이해하기 쉬움

#### 2. 작은 단위로 반복 (Small Iterations)
- 한 번에 하나의 Iteration만 진행
- 전체를 다 시각화하지 않음 (인지 부하 증가)
- Iteration 1: 인프라 → Iteration 2: 애플리케이션

#### 3. 빠른 피드백 루프 (Fast Feedback Loop)
- 각 Iteration마다 Checkpoint (검토 시점)
- 사용자가 확정하기 전까지 구현 안 함
- 잘못된 방향을 조기에 차단

#### 4. 검증 후 구현 (Validate Before Implementation)
- Checkpoint 통과 후에만 구현 시작
- 다이어그램과 실제 구현이 일치
- 사후 문서화가 아닌 사전 검증

### Iteration 구조

```
Sprint N
├── Sprint 시작: 프로젝트 방향 결정 (ADR 작성)
├── Iteration 1: {목표}
│   ├── 1. 시각화 (다이어그램, 필요시 ADR)
│   ├── 2. 🔍 Checkpoint (사용자 검토 및 확정)
│   ├── 3. 구현
│   └── 4. 검증
├── Iteration 2: {목표}
│   ├── 1. 시각화
│   ├── 2. 🔍 Checkpoint
│   ├── 3. 구현
│   └── 4. 검증
└── Iteration N: 통합 및 문서화
    ├── 전체 시스템 시각화
    └── README 작성
```

### ADR 작성 시점

ADR은 별도 작업이 아닌, **아키텍처 고민이 발생한 시점에 즉시 작성**합니다:

- **Sprint 시작 시:** 프로젝트 전체 방향 결정 (ADR-001, 002, 003)
- **인프라 시각화 시:** 기술 스택 선택 (ADR-004: MySQL vs PostgreSQL)
- **애플리케이션 시각화 시:** 아키텍처 패턴 선택 (ADR-005: Layered vs Hexagonal)

## Consequences

### 긍정적 영향

1. **잘못된 방향 사전 차단**
   - 코드 작성 전에 다이어그램으로 검토
   - Checkpoint에서 방향 수정 가능
   - 다시 작성하는 비용 최소화

2. **인지 부하 최소화**
   - 한 번에 하나의 Iteration만 집중
   - 작은 단위로 완결되는 작업
   - 전체를 한꺼번에 이해할 필요 없음

3. **빠른 피드백**
   - 각 Iteration마다 검토 시점
   - 빠르게 완결, 빠르게 검증
   - 긴 작업 기간 없이 지속적 진행

4. **문서화 자동 완성**
   - 다이어그램이 먼저 만들어짐
   - 구현과 문서가 일치 (싱크 문제 없음)
   - README에 바로 링크 가능

5. **의사결정 기록 자동화**
   - ADR이 자연스럽게 작성됨
   - "왜 이 기술을 선택했는가" 명확히 기록
   - 면접 시 근거 설명 가능

### 부정적 영향

1. **추가 작업 느낌**
   - 다이어그램 먼저 그리는 게 번거로울 수 있음
   - **완화:** AI가 다이어그램 생성, 사람은 검토만

2. **속도 저하 우려**
   - 바로 구현하는 것보다 느릴 수 있음
   - **완화:** 잘못된 방향으로 구현 후 다시 작성하는 것보다 빠름

3. **유연성 감소**
   - 다이어그램 확정 후 변경이 어려울 수 있음
   - **완화:** Checkpoint에서 충분히 검토, 필요 시 수정 가능

### 예시: Sprint 0

**Iteration 1: 인프라 환경 구축**
1. 시각화: Docker Compose 다이어그램 + ADR-004 작성
2. 🔍 Checkpoint 1: 사용자 검토 및 확정
3. 구현: Docker Compose 파일 작성
4. 검증: `make up` 실행 확인

**Iteration 2: 프로젝트 스캐폴딩**
1. 시각화: 패키지 구조 다이어그램 + ADR-005 작성
2. 🔍 Checkpoint 2: 사용자 검토 및 확정
3. 구현: Spring Boot 프로젝트 생성
4. 검증: ArchUnit 테스트 통과

### 핵심 원칙

> **"AI 시대에는 구현보다 시각화가 빠르다."**
>
> 사람은 코드보다 다이어그램이 이해하기 쉽다. AI에게 다이어그램을 먼저 그리게 하고, 사람이 검토한 후 구현하는 것이 효율적이다.

> **"작은 단위로 반복하면 인지 부하가 줄어든다."**
>
> 전체를 한꺼번에 시각화하면 오히려 복잡하다. Iteration별로 하나씩 시각화 → 검토 → 구현 반복하면 집중도가 높아진다.

> **"Checkpoint는 비용이 아니라 투자다."**
>
> 잘못된 방향으로 구현한 후 다시 작성하는 비용보다, Checkpoint에서 검토하는 시간이 훨씬 적다.
