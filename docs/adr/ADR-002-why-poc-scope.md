# ADR-002: 왜 PoC 범위로 축소했는가?

**날짜:** 2026-01-06
**상태:** Accepted

## Context

### 초기 계획의 문제점

**계획 1차: 대규모 데이터 처리 = MSA**
- ❌ What(대규모 데이터 처리가 무엇?)을 정의하지 않음
- ❌ How(MSA)부터 시작
- ❌ MSA는 도메인 분리지 대규모 데이터 처리가 아님

**계획 2차: 3가지 기술 모두 구현**
- ❌ 경쟁 상태 + 조회 최적화 + EDA
- ❌ 3가지를 다 하려다 보니 각각이 얕아질 위험
- ❌ 범위가 너무 큼 (실제로 필요한 것이 무엇인지 불명확)

**계획 3차: time-deal-service (모듈러 모놀리스)**
- ❌ 예상 기간: 3-4달 (제약 조건: 1-2달)
- ❌ 목적과 범위 불일치: 토이 프로젝트인데 MVP 수준 범위
- ❌ 완성 가능성 낮음: 헥사고날 학습 + Kafka 셋업 시간 과다
- ❌ "넓지만 얕음" 위험: 4개 Sprint를 다 하려다 각각이 얕아짐

### 제약 조건

1. **시간:** 1-2달 (길게 할 생각 없음)
2. **목적:** 이직용 기술 검증 토이 프로젝트
3. **타겟:** 시니어 백엔드 엔지니어 (네카라쿠배 + 핀테크)
4. **증명 방식:** GitHub + 블로그 (정량 지표)

### 선택지

**Option 1: MVP 범위 (3가지 기술 + 복잡한 아키텍처)**
- 범위: 경쟁 상태 + 조회 최적화 + EDA
- 아키텍처: 헥사고날 + 멀티 모듈 + Kafka
- 예상 기간: 3-4달
- 위험: 미완성 가능성 높음, 각 기술이 얕아짐

**Option 2: PoC 범위 (동시성 제어 하나만 깊게)**
- 범위: 재고 차감 동시성 제어 4가지 방법 비교
- 아키텍처: 레이어드 아키텍처 (단순)
- 예상 기간: 1-1.5달
- 장점: 완성 가능, 전문성 증명, 정량 지표

## Decision

**PoC (Proof of Concept) 범위로 축소합니다.**

### 최종 범위

**목표:**
> "재고 차감 동시성 제어 4가지 방법 성능 비교"

**포함:**
- 단일 도메인: Stock (재고) 관리만
- 단일 기능: 재고 차감 (데이터 정합성 보장)
- 4가지 구현: Pessimistic, Optimistic, Redis Lock, Lua Script
- 부하 테스트: k6로 TPS, Latency, Success Rate 측정
- 정량 비교: 성능 비교 표 + 그래프

**제거:**
- ❌ 멀티 모듈 (Product, Order, Payment 분리) → 단일 모듈
- ❌ 헥사고날 아키텍처 → 레이어드 아키텍처
- ❌ Kafka/EDA → 동시성 검증에 방해됨
- ❌ 조회 최적화 → 동시성 제어에 집중
- ❌ 복잡한 도메인 로직 → 최소한의 비즈니스 규칙

**유지:**
- ✅ DDD (Stock Aggregate)
- ✅ ArchUnit (아키텍처 규칙 강제 - 가성비 좋음)
- ✅ JUnit 5 (단위 테스트)

## Consequences

### 긍정적 영향

1. **완성 가능성 향상**
   - 1-1.5달 안에 완성 가능
   - 시간 제약 조건 충족

2. **전문성 증명**
   - 동시성 제어 하나를 극도로 깊게
   - "넓고 얕게" < "좁고 깊게"

3. **정량 지표 확보**
   - 4가지 방법 성능 비교
   - TPS, Latency, Success Rate 측정
   - 블로그 포스팅 자료

4. **면접 어필**
   - "이런 조건에서는 이 방법이 X% 빠릅니다"
   - 데이터 기반 기술 선택 근거 설명

5. **확장 가능성 유지**
   - Sprint 5: 조회 최적화 추가 (선택)
   - Sprint 6: EDA 추가 (선택)
   - 하나를 완벽하게 완성한 후 확장

### 부정적 영향

1. **좁은 범위**
   - 조회 최적화, EDA는 다루지 않음
   - **완화:** 확장 가능성 유지, 필요 시 추가 가능

2. **단순한 아키텍처**
   - 헥사고날, MSA는 보여주지 못함
   - **완화:** 아키텍처보다 "문제 해결 능력"이 중요

3. **MVP가 아닌 PoC**
   - 실제 서비스 수준은 아님
   - **완화:** 토이 프로젝트는 PoC가 적합, 기술 검증이 목적

### 핵심 교훈

> **"토이 프로젝트는 MVP가 아니라 PoC다."**
>
> 완성 가능한 작은 범위에서 기술적 깊이를 극대화하는 것이 이직용 포트폴리오의 핵심이다.

> **"대규모 트래픽 처리"라는 모호한 목표는 존재하지 않는다.**
>
> "재고 차감 시 동시성 제어를 4가지 방법으로 검증하고 성능을 비교한다"처럼 구체적으로 정의해야 올바른 범위 산정이 가능하다.
