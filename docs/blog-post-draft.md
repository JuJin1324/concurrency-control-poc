# 재고 차감 시나리오로 풀어보는 동시성 제어: 4가지 전략 완벽 비교 가이드

> **"대규모 트래픽 상황에서 데이터 정합성을 어떻게 보장할 것인가?"**
> 본 포스팅에서는 이커머스에서 가장 흔하지만 까다로운 '재고 차감' 문제를 해결하기 위한 4가지 동시성 제어 전략을 심층 분석하고, 정량적 테스트 결과를 바탕으로 실무 가이드를 제시합니다.

---

## 1. 문제 정의: Race Condition (경합 상태)

재고가 100개인 상품에 대해 1,000명의 유저가 동시에 '구매' 버튼을 눌렀을 때, 최종 재고가 0이 아닌 마이너스가 되거나 수량이 남는 현상을 본 적이 있으신가요? 

이것이 바로 **Race Condition(경합 상태)**입니다. 여러 스레드가 동시에 같은 공유 자원에 접근하여 수정을 시도할 때, 원자성(Atomicity)이 깨지면서 데이터 부정합이 발생합니다.

이 프로젝트에서는 이 문제를 해결하기 위해 가장 많이 쓰이는 4가지 방법을 선정하여 비교 실험을 진행했습니다.

---

## 2. 해결을 위한 4가지 제어 전략

### 1) Pessimistic Lock (비관적 락)
- **원리:** 데이터베이스의 `SELECT ... FOR UPDATE` 구문을 사용하여 레코드 자체에 락을 겁니다.
- **특징:** 데이터 충돌이 빈번할 때 안정적이지만, 락 점유 시간 동안 다른 요청이 대기하므로 성능 저하가 발생할 수 있습니다.

### 2) Optimistic Lock (낙관적 락)
- **원리:** JPA의 `@Version` 컬럼을 이용해 수정 시점에 버전 일치 여부를 확인합니다.
- **특징:** 락을 직접 걸지 않아 가볍지만, 충돌 발생 시 애플리케이션 레벨에서 재시도(Retry) 로직을 직접 구현해야 합니다.

### 3) Redis Distributed Lock (분산 락)
- **원리:** Redisson 라이브러리를 사용하여 Redis에 락을 획득/해제하는 논리적 락을 구현합니다.
- **특징:** DB의 물리적 락 비용을 줄일 수 있고 여러 대의 서버(분산 환경)에서 유효하지만, 락 획득/해제 자체의 네트워크 오버헤드가 존재합니다.

### 4) Redis Lua Script (원자적 연산)
- **원리:** Redis 내부에서 Lua 스크립트를 실행하여 '조회-비교-차감' 과정을 하나의 원자적 트랜잭션으로 처리합니다.
- **특징:** 싱글 스레드로 동작하는 Redis의 특성을 극대화하여 락 없이도 최상의 성능을 냅니다.

## 3. 성능 테스트 결과: 데이터로 증명하기

실제 환경과 유사한 부하를 주기 위해 **k6**를 활용하여 세 가지 시나리오를 테스트했습니다. (테스트 환경: Mac Studio M1 Max)

### 시나리오 1: Hell Test (선착순 이벤트)
*재고 100개 vs 5,000명 동시 접속 (극한의 경합 상황)*

| 제어 방식 | Latency (p95 / Avg) | Throughput (TPS) | 정합성 | 결과 요약 |
| :--- | :---: | :---: | :---: | :--- |
| **Lua Script** | **1.12s / 447ms** | **3,518 req/s** | ✅ | **압도적 1위.** 선착순 이벤트에 최적화 |
| **Optimistic** | 1.33s / 615ms | 3,354 req/s | ✅ | **의외의 선전.** 실패 시 즉시 리턴으로 효율 확보 |
| **Pessimistic** | 3.58s / 1.48s | 1,280 req/s | ✅ | **안정적이지만 느림.** 락 대기 오버헤드 가시화 |
| **Redis Lock** | 5.82s / 2.15s | 742 req/s | ✅ | **병목 발생.** 5,000개 락 요청 처리에 한계 |

### 시나리오 2: Extreme Load (시스템 한계 탐색)
*재고 10,000개 vs 지속적 부하*

- **Pessimistic Lock (p95 4.05ms):** 극한의 경합에서도 락 큐를 통한 순차 처리가 가장 안정적인 꼬리 지연 시간(Tail Latency)을 보여줌.
- **Lua Script (Avg 1.99ms):** 평균 응답 속도 면에서는 여전히 독보적인 성능 유지.

## 💡 핵심 인사이트 및 실무 가이드

### 1. 선착순 이벤트라면 고민 없이 'Lua Script'
재고 차감 로직이 비교적 단순하고 Redis 활용이 가능하다면, Lua Script가 가장 적은 리소스로 최고의 처리량(TPS)을 뽑아냅니다. 

### 2. 충돌이 많다고 낙관적 락이 무조건 느린 것은 아니다
테스트 결과, 낙관적 락은 충돌 시 즉시 리턴하는 'Fast Fail' 특성 덕분에 시스템 전체의 응답성을 지키는 데 유리했습니다. 다만, 재시도 로직이 너무 길어지면 오히려 독이 될 수 있습니다.

### 3. Redis Lock의 병목을 주의하라
분산 락은 만능이 아닙니다. 락을 얻기 위해 대기하는 스레드가 많아지면 톰캣 스레드 풀이 고갈되어 시스템 전체가 마비될 수 있습니다. (Hell Test에서 Max VU 5,000 기록)

### 4. Warm-up은 선택이 아닌 필수
JVM JIT 컴파일과 커넥션 풀 초기화 유무에 따라 지연 시간이 약 30% 이상 차이납니다. 실제 서비스 배포 시에도 적절한 Warm-up 시나리오가 필요함을 확인했습니다.
