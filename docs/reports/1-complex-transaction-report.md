# 📊 [Scenario 1] Complex Transaction Test Report

## 🎯 테스트 목적

재고 차감, 포인트 사용, 주문 이력 생성이 결합된 **복합 트랜잭션(Complex Transaction)** 상황에서 각 동시성 제어 방식의 데이터 정합성 보장 능력과 실질적 처리량을 검증한다.

**핵심 질문 (Key Questions):**
- "긴 트랜잭션(100ms+) 상황에서 낙관적 락의 재시도가 실효성이 있는가?"
- "비관적 락은 락 대기 시간에도 불구하고 낙관적 락보다 더 많은 성공 건수를 보장하는가?"
- "충돌 발생 시 모든 연관 데이터(재고, 포인트, 이력)가 원자적으로 롤백되는가?"

### 1.3 상세 테스트 시나리오 (Detailed Scenario)
이 테스트는 **가장 가혹한 경합 상황(Hotspot)**을 시뮬레이션하기 위해 다음과 같이 설계되었습니다.

1.  **집중 경합 (Hotspot):** 모든 가상 사용자(50 VUs)가 **동일한 상품(ID:1)**과 **동일한 사용자(ID:1)**의 포인트에 접근합니다.
2.  **긴 트랜잭션 (Long-lived Transaction):** 각 요청은 트랜잭션 진입 후 DB 업데이트 직전에 **100ms의 인위적인 지연**(`Thread.sleep`)을 가집니다. 이는 포인트 서비스 호출이나 복잡한 계산 로직을 시뮬레이션합니다.
3.  **다중 엔티티 수정:** 하나의 트랜잭션 내에서 `Stock`, `Point`, `OrderHistory` 3개 테이블에 대한 쓰기 연산이 동시에 발생합니다.
4.  **부하 모델:** `constant-vus` 에디터를 사용하여 50명의 사용자가 15초 동안 **틈 없이(No-think time)** 요청을 쏟아붓습니다.

## 💡 분석 및 결론 (Insights)

### 1. 전략적 추론: "비관적 락(Pessimistic Lock)의 데이터 정합성 보장 능력"
본 실험을 통해 우리는 **"비관적 락은 DB 엔진 수준에서 트랜잭션을 직렬화하여 데이터 정합성을 가장 확실하게 보장하는 수단"**이라는 아키텍처적 통찰을 얻었으며, 이는 향후 유사 시나리오에서도 핵심적인 설계 원칙이 될 것으로 예상됩니다.
- **관찰:** 낙관적 락이 외부의 도움(재시도 로직 등) 없이는 고경합 상황에서 무력하게 무너지는 것과 달리, 비관적 락은 추가적인 로직 없이도 모든 요청을 정합성에 맞게 처리했습니다.
- **경험적 체득:** 복잡한 비즈니스 로직이 얽힌 상황에서는, 여러 기술을 조합하는 것보다 DB Row Lock을 통해 정합성을 강제하는 것이 훨씬 단순하고 확실한 전략임을 증명했습니다.

### 2. 안정성이 곧 실질적 처리량 (Throughput)
- **관찰:** 응답 속도가 가장 느린 비관적 락(Avg 4.87s)이 최종 성공 건수(180건)에서 낙관적 락(156건)을 앞지름.
- **원인:** 낙관적 락은 높은 경합 상황에서 대부분의 연산이 실패와 재시도로 낭비(Wasted Work)되지만, 비관적 락은 락 대기를 통해 확실하게 한 건씩 완수하기 때문.
- **결론:** 경합이 심한 복합 트랜잭션에서는 **"차라리 줄을 세우는 것이 전체 시스템 효율에 더 좋다."**

### 3. "성공률 100%"의 진짜 의미: 자원 낭비 제로 (Zero Wasted Work)
- **의문:** 비관적 락의 총 요청(180건)이 낙관적 락(1,380건)보다 훨씬 적은데, 성공률 100%가 의미 있는 수치인가?
- **해석:** 비관적 락의 총 요청이 적은 이유는 **락 대기로 인해 요청 자체가 천천히 처리**되기 때문입니다. 이는 약점이 아니라 핵심 강점입니다. DB 커넥션을 획득한 모든 트랜잭션이 반드시 성공하므로, **커넥션을 허투루 소모한 적이 단 한 번도 없습니다.**
- **반면 낙관적 락:** 1,380건 중 1,242건은 커넥션을 잡고 100ms 동안 비즈니스 로직을 수행한 뒤, 마지막 버전 체크에서 충돌이 감지되어 전부 버려지는 **순수 매몰 비용**입니다. 이 1,242건이 점유한 DB 커넥션 시간은 시스템 자원을 소모했을 뿐 비즈니스 가치를 전혀 만들어내지 못했습니다.
- **결론:** 성공률은 단순히 "높으면 좋은 수치"가 아니라, **DB 커넥션 1회 사용당 비즈니스 성과를 얼마나 만들어냈는가(자원 효율성)**를 나타내는 지표입니다. 비관적 락의 100%는 "적게 시도해서 높은 것"이 아니라, **"자원 낭비 없이 가장 많은 비즈니스 성과를 낸 것"**입니다.

### 4. 재시도의 역설 (The Paradox of Retries)
- **관찰:** 낙관적 락에 재시도를 3회 추가하자 성공률은 4배 늘었으나, 응답 시간은 5배 이상 증가함.
- **원인:** 50명의 VUs가 100ms씩 점유하는 좁은 문을 통과하기 위해 여러 번 시도하는 과정에서, DB 커넥션 점유 시간만 늘어나고 병목이 심화됨.
- **결론:** 낙관적 락의 재시도는 충돌이 드문 환경(1-5%)에서만 유효하며, 고경합 환경에서는 **시스템 부하만 가중시키는 독**이 될 수 있음.

### 5. 자원 효율성 및 매몰 비용 (Resource Efficiency)
- **관찰:** HikariCP Max Pool Size가 10으로 제한된 환경에서 낙관적 락의 처리 효율이 급감함.
- **원인:** 낙관적 락은 트랜잭션 종료 시점에 충돌을 감지하므로, 실패하는 요청들이 커넥션을 점유하는 시간(100ms + 재시도 대기)은 시스템 입장에서 완전한 **매몰 비용(Wasted Work)**이 됨. 반면 비관적 락은 대기가 발생할지언정 커넥션을 점유한 동안은 확실한 성공을 보장함.
- **결론:** 자원이 제한적인 환경(Low Resource)일수록, 실패 확률이 높은 낙관적 락보다 비관적 락이 시스템의 전체적인 자원 효율성(Efficiency) 측면에서 유리함.

### 6. 낙관적 락의 빠른 실패를 무기로: 비공개 API에서의 가능성
- **관찰:** 낙관적 락(No-Retry)은 90%가 실패했지만, 응답 시간은 p95 0.48s로 비관적 락(5.68s)의 **12분의 1** 수준이었습니다.
- **공개 API에서는 부적절:** 불특정 다수의 사용자에게 "실패했으니 다시 시도하세요"라는 응답을 90% 확률로 돌려주는 것은 UX 관점에서 치명적입니다.
- **비공개 API에서는 무기가 됨:** 사내 시스템처럼 **클라이언트가 정해져 있는 비공개 API**라면 이야기가 달라집니다. 서버는 빠른 거절(Fail-fast)로 자원을 보호하고, 클라이언트는 **지수 백오프(Exponential Backoff) + 지터(Jitter)**를 적용한 스마트 재시도로 부하를 시간축으로 분산시킬 수 있습니다. 이 서버-클라이언트 협업 구조는 비관적 락이 서버 혼자 모든 부하를 떠안는 것보다 시스템 전체의 처리량을 끌어올릴 수 있는 잠재력을 가집니다.
- **핵심 전제:** 이 전략이 유효하려면 클라이언트의 재시도 정책을 서버 팀이 직접 제어하거나, 최소한 합의할 수 있어야 합니다. 즉, **API의 소비자가 통제 가능한 범위 내에 있는가**가 이 전략의 채택 기준입니다.

## 🚀 확장 전략 (Scalability Strategy)

*   **비관적 락 최적화:** 락 범위를 최소화하기 위해 트랜잭션 내부에서 I/O(외부 API)를 제거하고 순수 DB 연산만 남김.
*   **분산 트랜잭션 고려:** 서비스 규모 확대 시 2PC 대신 **Saga 패턴** 등을 통해 트랜잭션을 분리하고, 각 단계에서 적절한 락 전략(예: 보상 트랜잭션)을 선택.
*   **Fail-fast + Smart Retry:** 비공개 API 환경에서는 낙관적 락의 빠른 실패를 활용하여 서버 자원을 보호하고, 클라이언트의 지수 백오프 재시도로 부하를 시간축으로 분산시키는 협업 모델 검토. (시나리오 3에서 Redis + Fail-fast 전략으로 실증됨)
*   **User Sharding:** 동일 사용자(`userId=1`)에 대한 집중 경합은 샤딩을 통해 물리적으로 분산시킬 수 없는 영역이므로, 비즈니스 레벨의 대기열(Redis Queue 등) 도입 검토.
