# 📊 [Scenario 2] Low Contention Update Test Report

## 🎯 테스트 목적

충돌 가능성이 낮은 분산 환경(Low Contention)에서 낙관적 락(Optimistic Lock)의 성능적 우위를 검증하고, 비관적 락과의 공정한 비교(동일 지연 시간)를 통해 실제 실무적 이득을 측정한다.

**핵심 질문 (Key Questions):**
- "동일한 비즈니스 지연(100ms) 환경에서 낙관적 락은 비관적 락보다 정말 빠른가?"
- "재시도 오버헤드를 포함한 낙관적 락의 실질 처리량은 비관적 락을 압도하는가?"
- "자원 제약(DB Pool 10) 상황에서 커넥션을 빨리 반납하는 구조가 얼마나 유리한가?"

1.  **수요 분산 (Distributed Access):** 총 100개의 상품에 100명의 사용자가 랜덤하게 접근하여 충돌 확률을 낮게 유지합니다.
2.  **지연 시간 동기화:** 비관적/낙관적 락 모두 트랜잭션 내/외부 로직 처리에 **100ms**를 소요하도록 설정하여 공정성을 확보합니다.

4.  **부하 모델:** 100 VUs, 30초 지속 요청.

## 💡 분석 및 결론 (Insights)

### 1. 전략적 추론: "Redis + 낙관적 락 조합"의 잠재력
본 실험은 낙관적 락이 효율적으로 동작하기 위해선 **"DB 진입 전 단계에서의 부하 분산 또는 제어"**가 반드시 필요하다는 사실을 확인하는 계기였습니다.
- **관찰:** 100개 상품으로 수요가 분산된 환경(Low Contention)은 사실상 상위 계층에서 트래픽이 이미 분산되어 DB 부하가 임계치 아래로 유지되는 상태를 시뮬레이션한 것과 같습니다.
- **경험적 체득:** 낙관적 락은 단독으로는 고경합 상황에서 취약하지만, Redis 분산 락 등으로 진입 유입량을 조절하고 DB에서 최종 확인하는 **'계층적 제어 체계'**의 일원으로 동작할 때 비로소 비관적 락을 능가하는 높은 처리량을 보여줄 것으로 예상됩니다.

### 2. 지연 시간이 짧을 때의 반전: 비관적 락의 재발견
- **관찰:** 지연 시간을 100ms로 맞추자 낙관적 락과 비관적 락의 TPS 차이가 **단 4%** 내외로 좁혀짐.
- **원인:** 100개 상품에 부하가 분산되어 있어 비관적 락의 '락 경합' 자체가 거의 발생하지 않았기 때문. 커넥션 풀 대기만 해결되면 비관적 락도 충분히 높은 성능을 냄.
- **결론:** 분산이 잘 된 환경(Low Contention)에서는 굳이 복잡한 낙관적 락을 쓰지 않아도 **비관적 락만으로 충분히 훌륭한 성능을 낼 수 있음.**

### 2. 커넥션 반납 전략의 유효성
- **관찰:** 낙관적 락(No-Retry)이 근소하게(TPS 90 vs 87) 앞선 이유는 Sleep 시간 동안 커넥션을 풀에 반납했기 때문임.
- **원인:** 제한된 10개의 커넥션을 100명이 더 빠르게 순환시키며 사용할 수 있었던 아키텍처적 이득.
- **결론:** 로직이 매우 길거나(예: 1초 이상) 커넥션이 극도로 부족한 상황이라면 낙관적 락의 '커넥션 효율성'이 빛을 발함.

### 3. 재시도 오버헤드와 성능 역전의 임계점
- **관찰:** 충돌률이 4%에 불과한 환경임에도 불구하고, 재시도 3회를 도입하자마자 낙관적 락의 TPS가 하락하며 비관적 락의 턱밑까지 추격당함.
- **핵심 통찰:** "낙관적 락이 비관적 락보다 빠르다"는 명제는 **"재시도 로직이 거의 작동하지 않을 때"**만 유효함. 단 한 번의 재시도라도 발생하는 순간, 낙관적 락은 [로직 재수행 + DB 커넥션 재획득 + 네트워크 RTT]라는 비용을 지불하게 되어 비관적 락의 효율성을 이기기 어려워짐.
- **결론:** 재시도가 빈번하게 예상되는 환경이라면, 낙관적 락의 '락 없음' 이점은 '재시도 비용'에 의해 완전히 상쇄됨. 이때는 오히려 처음부터 줄을 세우는 비관적 락이 성능적으로 더 우월한 선택이 될 수 있음.

## 🚀 확장 전략 (Scalability Strategy)

*   **Simple is Best:** 저경합 환경에서는 구현이 단순하고 안정적인 **비관적 락을 기본 전략**으로 채택하되, 트래픽이 임계치를 넘어 DB 커넥션 풀 고갈이 예상될 경우 **Redis + 낙관적 락 체계**로의 전환을 검토해야 합니다.
*   **Long-running Task 분리:** 트랜잭션 내에서 시간이 오래 걸리는 작업(지연)이 있다면, 이를 트랜잭션 외부로 빼내어 커넥션 점유 시간을 줄이는 것이 어떤 락 전략보다 효과적입니다.
*   **계층적 제어:** Redis 분산 락으로 DB 진입 유입량을 제어하고, 낙관적 락으로 최종 정합성을 보장하는 구조는 대규모 분산 환경에서 가장 확장성이 뛰어난 모델이 될 것입니다.
