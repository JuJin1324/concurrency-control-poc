# 📊 Best Fit Scenario 2: Low Contention Update Test Report

## 🎯 테스트 목적

충돌이 드문 저경합 환경에서 **낙관적 락(Optimistic Lock)**이 비관적 락(Pessimistic Lock) 대비 가지는 **자원 점유 효율성**과 **시스템 확장성**의 차이를 확인한다. 기술의 단순 성능(TPS) 비교를 넘어, 동시성 제어 방식이 인프라 자원(CPU, Connection) 사용 패턴에 미치는 영향을 분석한다.

### 1.2 Tuning Config
#### [공통 설정]
*   **Virtual Threads:** `spring.threads.virtual.enabled: true`를 통한 동시 요청 처리 기반 구축.
*   **Business Logic:** `Thread.sleep(1000)`으로 외부 I/O 연동 등 실무적인 트랜잭션 지연 상황 시뮬레이션.

#### [낙관적 락 (Optimistic)]
*   **Transaction 전략:** `StockServiceHelper`를 통한 트랜잭션 분리 적용. 조회 후 커넥션을 반납하고, 로직 수행(1s) 후 수정 시점에 커넥션을 재획득하는 구조.
*   **Retry 전략:** `@Retryable` (Max 10, Exponential Backoff 50ms~1000ms).

#### [비관적 락 (Pessimistic)]
*   **Transaction 전략:** 단일 트랜잭션 내에서 `SELECT ... FOR UPDATE`로 락 획득 후 로직 종료 시까지 커넥션 및 로우 락 유지.

## 🧠 엔지니어링 여정과 기술적 깨달음 (Technical Journey)

### 1단계: 환경적 편향성과 원인 분석
초기 테스트에서 낙관적 락이 비관적 락에 비해 열세인 결과를 확인했습니다. 이는 로컬 환경의 낮은 네트워크 지연과 DB 락 오버헤드로 인해, **낙관적 락의 재시도 비용이 비관적 락의 대기 비용보다 상대적으로 높게 측정되는 환경적 특성**에 기인함을 발견했습니다.

### 2단계: 워크로드 시프팅(Workload Shifting)의 관점
낙관적 락의 핵심은 단순 성능 향상이 아니라 **DB의 부하를 애플리케이션 레이어로 분산시키는 전략**에 있음을 확인했습니다. 이는 확장이 어려운 DB의 수직 확장 대신, 수평 확장이 용이한 애플리케이션 서버를 활용하여 전체 시스템 동시성을 감당할 수 있는 설계적 기반을 제공합니다.

### 3단계: 아키텍처 이론과 물리적 환경의 간극 확인
낙관적 락의 트랜잭션을 분리하여 이론적 자원 점유를 줄였음에도 TPS가 비약적으로 늘지 않는 현상을 목격했습니다. 이를 통해 **설계 패턴의 적용만큼이나 인프라(커넥션 풀, 스레드 모델 등)의 물리적 한계점이 전체 시스템 스루풋에 결정적인 영향을 미친다**는 사실을 확인했습니다.

## 💡 분석 및 결론 (Insights)

### 1. 자원 관리의 철학: "워크로드 시프팅(Workload Shifting)"
본 실험을 통해 얻은 가장 핵심적인 깨달음은 낙관적 락의 가치가 단순히 '가벼움'에 있는 것이 아니라, **병목의 위치를 제어 가능한 영역으로 전이시키는 전략**에 있다는 점입니다. 
- **비관적 락**은 동시성 제어의 모든 부하를 DB 내부의 락 매니저가 담당하게 합니다. 이는 부하 증가 시 확장이 어렵고 비용이 비싼 DB CPU 자원을 소모하며, 시스템 전체의 스케일업(Scale-up)을 강제하는 원인이 됩니다.
- 반면, **낙관적 락**은 경합 처리의 부담을 애플리케이션 레이어의 재시도(Retry) 로직으로 끌어올립니다. 이는 DB 자원을 보존하는 대신, 상대적으로 확장(Scale-out)이 용이하고 저렴한 애플리케이션 서버의 CPU를 대신 활용하는 전략입니다. 즉, 낙관적 락은 인프라 구조를 'DB 중심'에서 'App 서버 중심'으로 유연하게 가져갈 수 있게 해주는 고도의 아키텍처적 선택지입니다.

### 2. 비즈니스 로직의 무게와 재시도 기회비용
실험 결과, 비즈니스 로직이 무거울수록(본 실험의 1초 지연 상황) 낙관적 락의 재시도 전략은 매우 높은 리스크를 수반함을 확인했습니다. 
- 낙관적 락에서 충돌이 발생하면, 단순히 쿼리를 다시 날리는 것이 아니라 **이미 수행한 1초의 값비싼 비즈니스 로직을 통째로 버리고 다시 처음부터 수행**해야 합니다. 이러한 '매몰 비용'은 경합이 발생하는 순간 시스템 전체의 유효 처리량을 급격히 떨어뜨리는 원인이 됩니다.
- 반면 비관적 락은 락 대기 시간 동안 CPU가 아무런 일도 하지 않고 '줄을 서서 기다리는' 형태를 취합니다. 로직이 무거울수록 무질서한 재시도(Retry)보다 **질서 있는 대기(Queueing)**가 시스템 전체의 자원 효율성 측면에서 훨씬 더 안정적이고 예측 가능한 결과를 만들어냅니다.

### 3. 커넥션 효율성과 시스템의 생존력
자원 한계 실험(10 Connections)을 통해 확인한 낙관적 락의 진정한 강점은 '최고 속도'가 아닌 **'수용 가능한 동시성 밀도'**에 있습니다.
- 비관적 락은 로직 수행 내내 커넥션을 점유해야 하므로, 커넥션 풀의 크기가 곧 시스템의 최대 동시 처리량(Parallelism)을 결정짓는 절대적인 벽이 됩니다. 
- 낙관적 락은 트랜잭션 분리 구조를 통해 로직 수행 중 커넥션을 풀에 반납함으로써, 동일한 수의 커넥션으로도 훨씬 더 많은 사용자의 요청을 수용할 수 있는 기반을 제공합니다. 비록 상위 계층의 병목으로 인해 TPS 수치 자체는 드라마틱하게 오르지 않더라도, **DB 레이어의 가용성을 극한으로 끌어올려 시스템의 완전한 마비를 방지**하는 생존 전략으로서의 가치를 증명했습니다.

### 4. 최종 아키텍처 전략 가이드

*   **낙관적 락(Optimistic Lock)이 유리한 상황:**
    데이터 업데이트 시 충돌 가능성이 낮고, 읽기 비중이 압도적인 서비스(예: 사용자 프로필 수정, 게시글 본문 수정 등)에 적합합니다. 특히 DB 자원을 아끼고 애플리케이션 서버를 수평 확장하여 부하를 감당하고자 하는 시스템에서 필수적인 전략입니다. 다만, 충돌 시 재시도 비용을 최소화하기 위해 비즈니스 로직을 가볍게 유지하는 것이 권장됩니다.

*   **비관적 락(Pessimistic Lock)이 유리한 상황:**

    재고 차감이나 결제처럼 쓰기 경합이 빈번하게 발생하고 데이터의 정확성이 최우선인 핵심 도메인에 적합합니다. 특히 비즈니스 로직이 무거워 재실행 시의 시간/자원 낭비가 큰 경우, DB의 강력한 안정적인 큐잉 메커니즘을 신뢰하여 예측 가능한 성능을 유지하는 것이 더 현명한 설계입니다.

## 🚀 확장 전략 (Scalability Strategy)



본 테스트 결과를 바탕으로, 트래픽 증가에 따른 시스템 확장 시 다음과 같은 전략을 제안한다.



### 1. 애플리케이션 수평 확장 (Scale-out)

낙관적 락을 채택한 서비스의 경우, DB 부하가 애플리케이션 레이어로 전이(Workload Shifting)되므로 **애플리케이션 서버 증설**만으로도 전체 처리량을 효과적으로 높일 수 있다. 가상 스레드와 비차단 I/O를 결합할 경우, 동일한 DB 자원으로도 수천 명 이상의 동시 접속자를 수용 가능하다.



### 2. 하이브리드 동시성 제어 (Hybrid Approach)

모든 도메인에 하나의 방식을 고집하기보다, 데이터의 특성에 따라 분리 적용한다.

- **Hot Data (인기 상품):** 경합이 치명적인 구간은 **비관적 락**으로 질서 있는 처리를 보장.

- **Normal Data (일반 프로필 등):** 충돌 확률이 낮은 대다수의 구간은 **낙관적 락**으로 DB 커넥션 풀을 보존.



### 3. DB 커넥션 풀 최적화 및 비동기 처리

낙관적 락의 자원 효율성을 극대화하기 위해, 트랜잭션 분리를 넘어 **메시지 큐(Kafka, Redis)를 통한 비동기 업데이트** 도입을 고려한다. 이를 통해 DB 직접 연결 시간을 0에 수렴하게 만들어 극단적인 스케일에서도 안정성을 확보할 수 있다.
