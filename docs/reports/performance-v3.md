# 📊 Concurrency Control Performance Report (V3)

## V2 → V3: 무엇이 바뀌었나

**V2 (Sprint 5-6):** 동일 조건에서 4가지 방식을 절대 비교 → "Lua Script가 압도적 1위"
**V3 (Sprint 7):** 각 방식이 가장 빛나는 시나리오를 설계하여 검증 → **"은탄환은 없다. 적재적소만 있을 뿐."**

V2에서는 DB Pool 50, 단일 상품이라는 자원 여유 환경에서 측정하여 Lua Script가 모든 지표에서 압도했습니다. 그러나 실무에서는 비즈니스 로직의 복잡도, 트래픽 대비 인프라 자원의 여유, API 소비자의 통제 가능 여부에 따라 최적 선택이 달라집니다. V3는 이를 4가지 시나리오로 증명합니다.

## 🏆 Executive Summary

> **비관적 락을 기본으로 시작하고, 인프라 자원이 부족해지는 시점에 Redis를 도입하라.**

| 시나리오 | 핵심 평가 축 | 우승자 | 핵심 인사이트 |
|:---|:---|:---|:---|
| **1. Complex Transaction** | 안정성 | **Pessimistic** | 복합 트랜잭션에서 자원 낭비 제로, 실질 처리량 1위 |
| **2. Low Contention** | 성능 | **Pessimistic** | 저경합에서도 종합적으로 최선 (TPS 차이 2.8%, 성공률 100%) |
| **3. Resource Protection** | 확장성 | **Redis + Optimistic** | 고부하(500 VUs)에서 DB 커넥션 보호, Fail-fast 전략 |
| **4. Extreme Performance** | 순수 성능 | **Lua Script** | 지연 없을 시 독보적 3,491 TPS, 단 비즈니스 로직 제약 |

**놀라운 발견:** 비관적 락의 강점을 증명하려 한 시나리오 1에서 낙관적 락의 Fail-fast 가능성을 발견했고, 낙관적 락의 강점을 증명하려 한 시나리오 2에서 비관적 락의 종합 우위를 확인했습니다. 테스트는 가설을 증명하는 것이 아니라, **가설 너머의 통찰을 발견하는 과정**이었습니다.

## 📊 시나리오별 상세 분석

### ① [Complex Transaction (상세보기)](1-complex-transaction-report.md)
*재고 차감 + 포인트 사용 + 주문 이력 생성이 결합된 복합 트랜잭션 (50 VUs, 15s)*

| 순위 | 방식 | TPS | p95 Latency | 성공률 (성공/총 요청) |
|:---:|:---|:---:|:---:|:---:|
| **1** | **Pessimistic** | 8.67 | 5.68s | **180 / 180 (100.0%)** |
| 2 | Optimistic (Retry 3) | 20.61 | 2.82s | 156 / 358 (43.6%) |
| 3 | Optimistic (No-Retry) | 89.26 | 0.48s | 138 / 1,380 (10.0%) |

> **Insight:** TPS가 가장 낮은 비관적 락이 실질 성공 건수에서 1위. 커넥션을 잡은 모든 트랜잭션이 반드시 성공하므로 **자원 낭비가 제로**입니다. 반면 낙관적 락의 1,242건 실패는 커넥션을 점유하고 100ms 로직을 수행한 뒤 버려지는 순수 매몰 비용입니다.
>
> **숨겨진 발견:** 낙관적 락의 빠른 실패(p95 0.48s)는 비공개 API 환경에서 클라이언트 협업(지수 백오프 + 지터) 기반의 부하 분산 전략으로 활용할 수 있는 잠재력을 가집니다.

### ② [Low Contention Update (상세보기)](2-low-contention-report.md)
*100개 상품에 100명이 분산 접근하는 저경합 환경 (100 VUs, 30s)*

| 순위 | 방식 | TPS | p95 Latency | 성공률 (성공/총 요청) |
|:---:|:---|:---:|:---:|:---:|
| **1** | Optimistic (No-Retry) | **89.36** | **1.18s** | 2,628 / 2,773 (94.8%) |
| **2** | **Pessimistic** | 86.95 | 2.04s | **2,706 / 2,706 (100.0%)** |
| 3 | Optimistic (Retry 3) | 83.50 | 1.33s | 2,600 / 2,600 (100.0%) |

> **Insight:** 낙관적 락이 설계상 유리한 저경합 환경에서도 비관적 락 대비 TPS 차이는 **단 2.8%**에 불과합니다. 재시도를 포함하면 오히려 비관적 락에 역전(83.50 vs 86.95)됩니다. 성능, 안정성, 구현 단순성을 종합하면 **저경합에서도 비관적 락이 최선의 선택**입니다.
>
> **숨겨진 발견:** 다만 낙관적 락은 같은 트래픽 대비 DB 커넥션을 아껴 쓴다는 본질적 가치가 있으며, 이 차이는 트래픽이 증가하여 커넥션 풀이 병목이 되는 시점에 결정적 차이를 만듭니다.

### ③ [Resource Protection (상세보기)](3-resource-protection-report.md)
*500 VUs가 5개 상품에 집중 접근하는 고부하 환경 (DB Pool 10, 100ms 지연)*

| 순위 | 방식 | TPS | p95 Latency | 성공률 (성공/총 요청) |
|:---:|:---|:---:|:---:|:---:|
| **1** | **Redis + Optimistic** | **154.28** | **3.29s** | 1,013 / 3,575 (28.3%) |
| 2 | Optimistic (No-Retry) | 92.21 | 5.33s | 920 / 2,344 (39.2%) |
| 3 | Pessimistic | 36.59 | 13.48s | 1,235 / 1,235 (100.0%) |

> **Insight:** 비관적 락의 100% 성공률이 무색하게, p95 13.48초는 비즈니스적으로 이미 실패한 요청입니다. Redis가 전방에서 유입량을 조절(Fail-fast)하고 낙관적 락이 최종 정합성을 보장하는 **계층적 방어 체계**가 시스템 가용성을 지키는 유일한 생존 전략입니다.
>
> **실무 원칙:** 비관적 락이 충분히 동작하는 상황에서 Redis를 섣불리 도입하지 마라. 관리 비용이라는 숨겨진 대가가 있다.

### ④ [Extreme Performance (상세보기)](4-extreme-performance-report.md)
*비즈니스 지연(100ms)을 제거한 순수 오버헤드 비교 (500 VUs, 20s)*

| 순위 | 방식 | TPS | p95 Latency | 성공률 (성공/총 요청) |
|:---:|:---|:---:|:---:|:---:|
| **1** | **Lua Script** | **3,491.50** | **0.29s** | 70,106 / 70,106 (100.0%) |
| 2 | Optimistic | 985.91 | 1.10s | 8,756 / 20,016 (43.7%) |
| 3 | Pessimistic | 978.00 | 1.01s | 19,847 / 19,847 (100.0%) |
| 4 | Redis Lock | 602.42 | 1.88s | 12,316 / 12,384 (99.4%) |

> **Insight:** Lua Script가 2위 대비 3.5배의 독보적 TPS를 기록했지만, 이 성능은 비즈니스 로직을 극단적으로 간소화한 전제 조건에서만 가능합니다. 같은 간소화를 비관적 락에 적용하면 시나리오 3 대비 **26.7배 향상**(36 → 978 TPS)을 얻을 수 있으므로, Lua Script 도입의 정당성은 TPS가 아니라 **DB 커넥션 자원 보호**에 있어야 합니다.
>
> **비관적 락의 반전:** 시나리오 3에서 최하위(36 TPS) → 시나리오 4에서 실질적 2위(978 TPS). 비관적 락 자체가 느린 것이 아니라, **락을 잡은 채 유지하는 비즈니스 로직 시간이 진짜 원인**이었음이 증명되었습니다.

## 🔑 시나리오 교차 분석 (Cross-Scenario Insights)

### 1. 인프라 자원과 Redis 도입의 상관관계

| 환경 | DB Pool | Redis 도입 효과 | 최적 선택 |
|:---|:---:|:---|:---|
| **V2 (Sprint 5-6)** | 50 | 자원 여유 → Redis는 네트워크 홉만 추가, 오히려 성능 저하 | Pessimistic > Redis Lock |
| **V3 (Sprint 7)** | 10 | 자원 부족 → Redis의 진가 발휘 | Lua > Redis+Optimistic > Optimistic > Pessimistic |

**핵심 교훈: 자원이 널널한 환경에서 Redis 도입은 부질없다.** 비관적 락만으로 충분한 상황에서 Redis를 끼워넣으면 네트워크 홉만 추가되어 오히려 성능이 떨어집니다. Redis 도입의 정당한 시점은 **트래픽 대비 인프라 자원이 부족해지기 시작할 때**입니다.

### 2. 비즈니스 로직 지연에 따른 순위 반전

| 방식 | 100ms 지연 (시나리오 3) | 0ms 지연 (시나리오 4) | 변화율 |
|:---|:---:|:---:|:---:|
| Pessimistic | 36 TPS (최하위) | 978 TPS (2위) | **27배 ↑** |
| Redis Lock | 154 TPS (1위) | 602 TPS (최하위) | 4배 ↑ |
| Lua Script | _(N/A)_ | 3,491 TPS (1위) | - |

비즈니스 로직에 지연이 있으면 Redis 계층의 보호가 빛을 발하고, 지연이 없으면 Redis 네트워크 RTT가 오히려 병목이 됩니다. **기술 선택은 비즈니스 로직의 복잡도에 의해 결정**되어야 합니다.

### 3. "낙관적 락이 비관적 락보다 빠르다"는 통념의 검증

| 시나리오 | 낙관적 락 TPS | 비관적 락 TPS | 결과 |
|:---|:---:|:---:|:---|
| 1. Complex Transaction (No-Retry) | 89.26 | 8.67 | 낙관적 ↑ (단, 성공 138건 vs 180건) |
| 1. Complex Transaction (Retry 3) | 20.61 | 8.67 | 낙관적 ↑ (단, 성공 156건 vs 180건) |
| 2. Low Contention (No-Retry) | 89.36 | 86.95 | **차이 2.8%** (단, 성공률 95% vs 100%) |
| 2. Low Contention (Retry 3) | 83.50 | 86.95 | **비관적 역전** |
| 3. Resource Protection | 92.21 | 36.59 | 낙관적 ↑ (고부하 환경) |
| 4. Extreme Performance | 985.91 | 978.00 | **차이 0.8%** (단, 성공률 44% vs 100%) |

**결론:** "낙관적 락이 빠르다"는 명제는 **"재시도가 없고, 실패를 허용할 때"**만 성립하는 조건부 명제입니다. 실무에서 실패를 허용하지 않는다면, 비관적 락이 성능까지 포함하여 종합적으로 우위에 있습니다.

## 🚀 Final Recommendation (도입 가이드)

> V2의 "Lua Script가 만능"이라는 결론을 V3는 수정합니다.
> **핵심 전략: 낙관적 락으로 시작하고, 모니터링으로 비관적 락을 걸어야 할 지점을 식별하라.**

### 0순위: Optimistic Lock (낙관적 락) — 전체 엔티티의 기본 보험

**적용 대상:** 모든 도메인 엔티티 (JPA `@Version`)

낙관적 락은 비관적 락보다 "빠르기 때문에" 쓰는 것이 아닙니다. 본 PoC가 증명했듯 성능 차이는 미미하거나 오히려 비관적 락이 우위입니다. **낙관적 락의 진짜 가치는 "모든 엔티티에 대한 경합 감지 보험"**입니다.

- 모든 엔티티에 `@Version`을 기본 적용하여, 혹시 모를 동시 수정 상황에서 데이터가 조용히 덮어써지는 것을 방지합니다.
- 평상시에는 성능 오버헤드가 거의 없으며(버전 컬럼 비교 1회), 충돌이 발생할 때만 `ObjectOptimisticLockingFailureException`이 발생합니다.
- 이 예외를 **모니터링(로그 집계, 알림)**하면, 경합이 빈번한 엔티티를 자연스럽게 식별할 수 있습니다.
- 식별된 엔티티에 대해서만 선택적으로 비관적 락을 적용하는 **점진적 최적화(Progressive Optimization)** 전략의 출발점이 됩니다.

```
[도입 흐름]
전체 엔티티 @Version 적용 (보험)
        ↓
낙관적 락 충돌 모니터링 (감지)
        ↓
경합 빈번 엔티티 식별 (분석)
        ↓
해당 엔티티만 비관적 락 적용 (최적화)
        ↓
트래픽 증가 시 Redis 계층 추가 (스케일)
```

### 1순위: Pessimistic Lock (비관적 락) — 경합 지점의 확실한 해결책

**적용 대상:** 낙관적 락 모니터링을 통해 경합이 식별된 엔티티

- 낙관적 락 충돌이 빈번하게 관측되는 엔티티에 대해 `SELECT ... FOR UPDATE`를 적용합니다.
- RDBMS만으로 데이터 정합성을 가장 확실하게 보장하며, 추가 인프라(Redis) 없이 운영 복잡도를 최소화합니다.
- 저경합~중경합 환경에서 낙관적 락과 TPS 차이가 미미하면서도 성공률 100%를 보장합니다.
- **전환 시점:** 트래픽 증가로 응답 시간이 비즈니스 허용 임계치를 넘어설 때.

### 2순위: Redis + Optimistic Lock (계층적 방어) — 스케일 전략

**적용 대상:** DB 커넥션 풀이 실질적 병목이 되는 고부하 환경

- Redis가 전방에서 유입량을 제어(Fail-fast)하고, 낙관적 락이 최종 정합성을 보장합니다.
- 서버의 빠른 거절 + 클라이언트의 스마트 재시도(지수 백오프 + 지터) 협업이 핵심입니다.
- **주의:** 비관적 락이 충분히 동작하는 환경에서 섣불리 도입하지 마십시오. Redis의 관리 비용은 숨겨진 대가입니다.

### 3순위: Lua Script (Redis as Primary) — 특수 전략

**적용 대상:** 단순 로직(단순 차감) + 극한 트래픽의 특정 Hotspot

- 비즈니스 로직이 단순하고, 초당 수천 건 이상의 극한 트래픽이 몰리는 특정 상품(예: 한정판 딜)에만 **선별적으로** 도입하십시오.
- 도메인 로직을 간소화하면 비관적 락만으로도 27배 TPS 향상이 가능하므로, 순수 TPS만으로는 도입이 정당화되지 않습니다.
- 도입 시 데이터 유실 대응 및 DB 동기화 전략(Kafka, CDC 등)은 필수입니다.
