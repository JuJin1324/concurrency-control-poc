# 📊 [Scenario 3] Resource Protection Test Report

## 🎯 테스트 목적

고부하(500 VUs) 상황에서 **비관적 락(Pessimistic Lock)** 단독 사용 시 발생하는 DB 커넥션 고갈 문제를 확인하고, **Redis 분산 락 + 낙관적 락 조합**을 통한 계층적 제어가 시스템의 생존력(Availability)을 어떻게 높이는지 검증한다.

**핵심 질문 (Key Questions):**
- "비관적 락은 왜 고부하 상황에서 시스템 전체의 응답 거부를 유발하는가?"
- "Redis 분산 락을 전계층에 배치했을 때 DB 커넥션 풀 보호 효과는 어느 정도인가?"
- "낙관적 락(Version)은 이 구조에서 어떤 역할을 수행하는가?"

## 📊 지표 해석 가이드 (Interpretation Guide)

1.  **시스템 생존력 (System Availability):**
    *   **의미:** 고부하 상황에서 시스템이 응답을 거부(Connection Refused)하지 않고 제어된 응답을 줄 수 있는 능력.
    *   **판단 기준:** 500 VUs 상황에서 HTTP 500(Timeout)이 발생하더라도 WAS/DB가 마비되지 않고 다른 API 응답이 가능한지 여부.

2.  **p95 Latency (Tail Latency):**
    *   **의미:** 상위 5% 사용자가 겪는 최대 지연 시간.
    *   **판단 기준:** 비관적 락은 커넥션 대기로 인해 이 지표가 무한히 발산할 수 있으며, Redis 조합은 락 타임아웃 설정에 의해 상한선이 제어됨.

## 💡 분석 및 결론 (Insights)

### 1. 전략적 추론: "자원 보호를 위한 계층적 방어의 필연성"

본 실험은 고부하 환경에서 시스템 전체의 생존을 위해선 **"DB 커넥션 점유와 락 대기 로직의 물리적 분리"**가 필수적임을 정량적으로 증명했습니다.

- **Redis의 '방화벽' 효과:** 낙관적 락 단독(No-Retry) 환경에서는 500명의 사용자가 한꺼번에 DB 커넥션을 요청했습니다. 결과적으로 p95 지연 시간이 5.33초까지 치솟았는데, 이는 **트랜잭션이 실패할 것임에도 불구하고 5초 동안 DB 커넥션을 붙잡고 있었다**는 뜻입니다. 반면 Redis 조합은 `waitTime(3s)`으로 이를 전방 차단하여 DB 커넥션 풀을 정교하게 보호했습니다.
- **비관적 락의 한계:** 모든 요청을 성공시키지만, 커넥션을 잡고 100ms씩 점유함에 따라 산술적으로 발생하는 거대한 대기열이 시스템 전체를 자원 고갈(Resource Exhaustion) 직전으로 몰아넣습니다.

### 2. Redisson WaitTime의 가치: "성공률과 생존력의 트레이드오프"

- **관찰:** Redis 조합은 높은 실패율에도 불구하고 처리량(TPS)은 비관적 락보다 4.2배 높고 지연 시간은 안정적으로 제어되었습니다.
- **핵심 통찰:** Redisson의 `waitTime`은 일종의 **백프레셔(Backpressure)**로 작동합니다. 고경합 상황에서 `waitTime`을 늘려 성공률을 높이려는 시도는 비관적 락의 병목을 Redis 계층으로 옮겨올 뿐입니다. 시스템 가용성을 위해선 서버가 **'빠른 거절(Fail-fast)'**을 수행하고, 완결성은 클라이언트의 재시도 정책으로 해결하는 것이 분산 시스템의 표준 전략입니다.

### 3. 가용성 격리(Availability Isolation)와 낙관적 락의 역할

- **분리된 대기열:** Redis 계층에서 대기열을 형성함으로써, 극심한 경합 상황에서도 DB 엔진 부하 및 커넥션 풀 상태가 안정적으로 유지됩니다. 락 획득 대기가 DB 진입 전(Redis 단계)에 발생하므로, 타임아웃 실패가 DB 자원에 영향을 주지 않습니다.
- **낙관적 락의 수비대 역할:** Redis 락이 유입량을 1차로 걸러주면, 낙관적 락(`@Version`)은 최종적으로 데이터 무결성을 검증하는 '최후의 수비대' 역할을 수행하여 계층적 방어 체계를 완성합니다.

### 4. 전략적 선택의 임계점: "재시도(Scenario 2) vs 빠른 실패(Scenario 3)"

- **저경합 (Scenario 2):** 충돌 확률이 낮으므로 **재시도를 통해 성공률 100%를 달성**하는 것이 UX 측면에서 우월합니다.
- **고경합 (Scenario 3):** 충돌 확률이 극도로 높아 재시도가 오히려 대기열을 폭증시키고 시스템을 마비시킵니다. 이때는 성공률을 희생하더라도 인프라를 보호하기 위한 **'빠른 실패'가 유일한 생존 전략**이 됩니다.

### 5. 엔지니어링 의사결정: "100% 성공은 과연 성공인가?"

본 PoC를 통해 도출된 최종적인 동시성 제어 선택 가이드는 다음과 같습니다.

1.  **기본 전략: 비관적 락(Pessimistic Lock)을 1순위로 채택하라**
    *   비관적 락은 가장 단순하며, RDBMS만으로 모든 요청을 누락 없이 처리할 수 있는 가장 강력한 안정성을 제공합니다. 외부 인프라(Redis)에 대한 의존성 없이 정합성을 보장한다는 것은 운영상 대체 불가능한 장점입니다.

2.  **비즈니스적 성공의 재정의 (The 13.48s Question)**
    *   비관적 락은 본 실험에서 100%의 성공률을 보였지만, 평균 10초, 최대 26초가 소요되었습니다. 여기서 우리는 자문해야 합니다. **"과연 13.48초(p95)가 걸려서 성공 처리된 요청을 비즈니스적으로 성공한 요청이라고 할 수 있는가?"**
    *   대부분의 현대적 서비스에서 10초 이상의 대기 시간은 사용자의 이탈과 타임아웃 에러를 의미합니다. 기술적으로는 DB에 데이터가 남았을지 몰라도, 비즈니스 관점에서는 이미 실패한 요청입니다.

3.  **계층적 제어(Redis + 낙관적 락)로의 전환 시점**
    *   따라서 무조건적인 성능 향상이 아니라, **비즈니스적으로 허용 가능한 레이튼시(Latency Threshold)를 먼저 설정**하십시오.
    *   비관적 락으로 운영하던 중, 트래픽 증가로 인해 응답 시간이 이 임계치를 넘어서기 시작할 때가 바로 Redis를 활용한 계층적 방어 체계를 도입해야 하는 시점입니다.

4.  **전환 후 운영: 서버와 클라이언트의 협업**
    *   성공적인 동시성 제어는 서버 단독의 과제가 아니라, 클라이언트와의 정교한 협업 결과물입니다.
    *   **서버의 책임 (Fail-fast):** 인프라 임계치를 넘어서는 요청을 `waitTime(3s)` 이내에 빠르게 거절하여 클라이언트에게 제어권을 넘깁니다.
    *   **클라이언트의 책임 (Smart Retry):** 거절 응답을 받은 클라이언트는 **지수 백오프(Exponential Backoff)**와 **지터(Jitter)**를 활용하여 재시도 시점을 분산시킵니다.
    *   이 협업이 완성될 때, 비관적 락의 '지연된 성공'보다 훨씬 높은 실질적 비즈니스 성공률을 달성할 수 있습니다.

---

**결론적으로, 고부하 상황에서는 "모든 요청의 100% 성공"이라는 관습적 목표가 오히려 시스템 전체를 위협할 수 있습니다. 이때는 "허용 범위 내의 레이튼시를 보장하는 성공"으로 목표를 전환하고, Redis와 낙관적 락을 결합한 계층적 방어 체계를 통해 시스템의 가용성을 지키는 것이 올바른 선택입니다.**

## 🚀 확장 전략 (Scalability Strategy)

*   **Fail-fast & Smart Retry:** 서버의 제어된 거절 응답을 기반으로 클라이언트에서 지수 백오프(Exponential Backoff) 등 스마트 재시도 전략을 결합하여 실제 사용자 성공률 보완.
*   **Adaptive Load Control:** 고정된 `waitTime` 대신 시스템 부하 상황에 따라 대기 시간을 동적으로 조절하는 적응형 부하 제어 알고리즘 검토.
*   **Multi-layered Defense:** Redis 분산 락으로 1차 진입 유입량을 제어하고, DB 낙관적 락(@Version)을 '최종 수비대'로 활용하여 데이터 무결성과 시스템 안정성을 동시에 확보하는 표준 아키텍처 수립.
