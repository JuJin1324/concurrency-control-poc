# 📊 [Scenario 3] Resource Protection Test Report

## 🎯 테스트 목적

고부하(500 VUs) 상황에서 **비관적 락(Pessimistic Lock)** 단독 사용 시 발생하는 DB 커넥션 고갈 문제를 확인하고, **Redis 분산 락 + 낙관적 락 조합**을 통한 계층적 제어가 시스템의 생존력(Availability)을 어떻게 높이는지 검증한다.

**핵심 질문 (Key Questions):**
- "비관적 락은 왜 고부하 상황에서 시스템 전체의 응답 거부를 유발하는가?"
- "Redis 분산 락을 전계층에 배치했을 때 DB 커넥션 풀 보호 효과는 어느 정도인가?"
- "낙관적 락(Version)은 이 구조에서 어떤 역할을 수행하는가?"

## 📊 지표 해석 가이드 (Interpretation Guide)

1.  **시스템 생존력 (System Availability):**
    *   **의미:** 고부하 상황에서 시스템이 응답을 거부(Connection Refused)하지 않고 제어된 응답을 줄 수 있는 능력.
    *   **판단 기준:** 500 VUs 상황에서 HTTP 500(Timeout)이 발생하더라도 WAS/DB가 마비되지 않고 다른 API 응답이 가능한지 여부.

2.  **p95 Latency (Tail Latency):**
    *   **의미:** 상위 5% 사용자가 겪는 최대 지연 시간.
    *   **판단 기준:** 비관적 락은 커넥션 대기로 인해 이 지표가 무한히 발산할 수 있으며, Redis 조합은 락 타임아웃 설정에 의해 상한선이 제어됨.

## 💡 분석 및 결론 (Insights)



### 1. 전략적 추론: "자원 보호(Resource Protection)를 위한 계층적 방어의 필연성"

본 실험은 고부하 환경에서 시스템 전체의 생존을 위해선 **"DB 커넥션 점유와 락 대기 로직의 물리적 분리"**가 필수적임을 증명하는 계기가 되었습니다.

- **관찰:** 비관적 락 단독 사용 시 성공률은 100%를 유지하지만, p95 응답 시간이 14초를 상회하며 인프라 자원이 임계치에 도달함.

- **원인:** 상품당 약 100명의 대기열(500 VUs / 5개 상품)이 형성된 상황에서, 각 트랜잭션이 커넥션을 잡고 100ms씩 점유함에 따라 산술적으로 **최소 10초 이상의 대기 시간(100ms * 100명)**이 발생함. 여기에 제한된 10개의 커넥션 풀을 얻기 위한 큐잉 지연이 중첩되어 14초대 지연이 형성됨. 이는 HikariCP의 기본 타임아웃(30s) 절반에 육박하는 수치로, 부하가 조금만 더 지속될 경우 시스템 전체가 응답을 거부하게 되는 **자원 고갈(Resource Exhaustion)** 직전의 상태임.

- **핵심 통찰:** 진정한 자원 보호는 모든 요청을 성공시키는 것이 아니라, 인프라가 감당 가능한 수준까지만 트래픽을 통과시키고 나머지는 격리된 계층(Redis)에서 제어하는 것임.



### 2. Redisson WaitTime의 가치: "성공률과 생존력의 트레이드오프"

- **관찰:** Redis 조합은 33%의 실패율이 발생했음에도 불구하고 처리량(TPS)은 비관적 락보다 1.8배 높고 지연 시간은 안정적으로 제어됨.

- **원인:** Redisson에 설정된 대기 시간(`waitTime`: 3s)이 일종의 **백프레셔(Backpressure)**로 작동하여, 처리가 불가능한 부하를 DB 진입 전 빠르게 거절(Fail-fast)했기 때문.

- **결론:** 고경합 상황에서 `waitTime`을 늘려 성공률을 높이려는 시도는 비관적 락의 병목을 Redis 계층으로 옮겨올 뿐임. 시스템 전체의 가용성을 위해선 서버가 **'빠른 거절'**을 수행하고, 완결성은 클라이언트의 재시도 정책으로 해결하는 것이 분산 시스템의 표준 전략임.



### 3. 가용성 격리(Availability Isolation)의 효과



- **관찰:** Redis 계층에서 대기열을 형성함으로써, 극심한 경합 상황에서도 DB 엔진 부하 및 커넥션 풀 상태가 안정적으로 유지됨.



- **원인:** 락 획득 대기가 DB 커넥션 요청 전(Redis 단계)에 발생하므로, 설령 타임아웃으로 실패하더라도 DB 자원에는 아무런 영향을 주지 않음.



- **주의사항 (SPOF 위험):** Redis 자체가 새로운 병목이 되거나 장애가 발생할 경우 시스템 전체가 마비되는 **단일 장애점(SPOF)**이 될 수 있음. 따라서 실운영 환경에서는 Redis 클러스터링(Clustering)이나 센티널(Sentinel)을 통한 고가용성 확보가 필수적이며, Redis 장애 시 비관적 락으로 전환하거나 즉시 거절 응답을 보내는 **폴백(Fallback) 전략**이 반드시 병행되어야 함.







### 4. 재시도(Scenario 2) vs 빠른 실패(Scenario 3)의 임계점







본 시나리오의 '빠른 실패' 전략은 시나리오 2의 '재시도' 전략과 상충하는 것이 아니라, 트래픽 경합도에 따른 **전략적 선택의 전환**입니다.







- **저경합 (Scenario 2):** 충돌 확률이 낮으므로 재시도 오버헤드가 적습니다. 이때는 **재시도를 통해 성공률 100%를 달성하는 것이 사용자 경험(UX) 측면에서 우월**합니다.







- **고경합 (Scenario 3):** 충돌 확률이 극도로 높아 재시도가 오히려 대기열을 폭증시키고 시스템을 마비시킵니다. 이때는 **성공률을 희생하더라도 인프라를 보호하기 위한 '빠른 실패(Fail-fast)'가 유일한 생존 전략**이 됩니다.







- **결론:** "재시도가 정답인가, 빠른 실패가 정답인가"에 대한 해답은 결국 **현재 시스템이 직면한 경합의 임계치**가 어디인가에 달려 있습니다.















### 5. 엔지니어링 철학: "기술 도입의 보수적 접근과 관리 비용"







계층적 제어(Redis + Optimistic)가 비관적 락보다 높은 처리량을 보이더라도, 이를 **1차 전략으로 성급하게 채택하는 것은 지양**해야 합니다.







- **RDBMS의 신뢰성:** 비관적 락은 RDBMS 벤더들이 수십 년간 최적화해온 가장 견고하고 검증된 동시성 제어 수단입니다. 외부 인프라 의존성 없이 DB 자체만으로 완결성을 가질 수 있다는 것은 운영상 매우 큰 이점입니다.







- **도입 목적의 순수성:** 단순히 기술적 호기심이나 무리한 인프라 비용 절감(RDBMS 스펙 하향)을 목적으로 Redis를 도입해서는 안 됩니다. Redis 역시 라이선스 비용뿐만 아니라 모니터링, 백업, 장애 대응 등 **'숨겨진 관리 비용(Management Cost)'**이 수반되는 별도의 인프라임을 인지해야 합니다.







- **도입 시점:** 비즈니스 로직 최적화와 DB 수직 확장(Scale-up)을 먼저 검토한 후, 그럼에도 불구하고 특정 핫스팟 자원에 의한 커넥션 풀 고갈이 실질적인 위협이 될 때 비로소 계층적 제어 도입을 검토하는 것이 올바른 순서입니다.

## 🚀 확장 전략 (Scalability Strategy)



*   **Fail-fast & Smart Retry:** 서버의 제어된 거절 응답을 기반으로 클라이언트에서 지수 백오프(Exponential Backoff) 등 스마트 재시도 전략을 결합하여 실제 사용자 성공률 보완.

*   **Adaptive Load Control:** 고정된 `waitTime` 대신 시스템 부하 상황에 따라 대기 시간을 동적으로 조절하는 적응형 부하 제어 알고리즘 검토.

*   **Multi-layered Defense:** Redis 분산 락으로 1차 진입 유입량을 제어하고, DB 낙관적 락(@Version)을 '최종 수비대'로 활용하여 데이터 무결성과 시스템 안정성을 동시에 확보하는 표준 아키텍처 수립.
