# k6: 현대적인 부하 테스트 및 성능 측정 가이드

본 문서는 동시성 제어 PoC의 성능 측정을 위해 채택한 **k6**의 핵심 개념, 철학 및 성능 측정 방법론을 정리합니다.

---

## 1. k6 개요 및 탄생 배경

### 1.1 k6란?
k6(구 LoadImpact)는 Grafana Labs에서 개발한 오픈 소스 부하 테스트 도구입니다. 개발자 친화적인(Developer-centric) 설계를 지향하며, Go 언어로 구현된 고성능 엔진과 JavaScript(ES6) 기반의 시나리오 작성 환경을 제공합니다.

### 1.2 탄생 배경 (왜 k6인가?)
기존의 부하 테스트 도구들(JMeter, Locust 등)은 다음과 같은 한계가 있었습니다:
- **JMeter:** GUI 기반 설정의 복잡성, XML 기반 구성의 버전 관리 어려움, 무거운 리소스 사용.
- **Locust:** Python의 GIL(Global Interpreter Lock)로 인한 단일 프로세스 성능 제한.
- **Gatling:** Scala 언어의 높은 진입 장벽.

k6는 이러한 문제를 해결하기 위해 **"코드로 정의하는 부하 테스트(Load Testing as Code)"**를 핵심 가치로 삼아 탄생했습니다.

## 2. 핵심 철학 및 도구 비교

### 2.1 k6의 핵심 철학
1. **코드 중심 (Scripting in JS):** UI가 아닌 코드로 시나리오를 작성하여 버전 관리(Git)와 협업이 용이합니다.
2. **자동화 및 CI/CD 친화적:** 명령행(CLI) 기반으로 동작하며, 통과 기준(Thresholds)을 통해 파이프라인에서 자동 성공/실패 판정이 가능합니다.
3. **고성능 & 저리소스:** Go 엔진을 사용하여 수천 개의 가상 사용자(VU)를 단일 머신에서 효율적으로 시뮬레이션합니다.

### 2.2 부하 테스트 도구 비교

| 특징 | k6 | JMeter | Gatling | Locust |
| :--- | :--- | :--- | :--- | :--- |
| **언어** | JavaScript (Go engine) | GUI / XML / Java | Scala / Java / Kotlin | Python |
| **성능** | 매우 높음 | 중간 | 높음 | 중간 (분산 필요) |
| **리소스 효율** | 우수 | 낮음 (JVM 오버헤드) | 중간 | 중간 |
| **학습 곡선** | 낮음 (JS) | 중간 (GUI 익숙해지면 낮음) | 높음 (Scala) | 매우 낮음 (Python) |
| **CI/CD 통합** | 매우 쉬움 | 중간 | 쉬움 | 중간 |

## 3. k6 핵심 개념

### 3.1 VU (Virtual User)
가상 사용자를 의미하며, 실제 사용자의 행동을 모방하는 독립적인 실행 루프입니다. 각 VU는 시나리오 코드를 병렬로 실행합니다.

### 3.2 RPS (Request Per Second) vs TPS (Transactions Per Second)
- **RPS:** 초당 서버에 전송되는 HTTP 요청 수.
- **TPS:** 초당 완료되는 비즈니스 트랜잭션(예: 재고 차감 전체 로직) 수.
- k6에서는 `http_req_rate` 메트릭을 통해 이를 측정합니다.

### 3.3 Metric (메트릭)
테스트 결과로 수집되는 데이터 포인트입니다.
- **Counter:** 누적 합계 (예: 에러 횟수)
- **Gauge:** 마지막 값 (예: 현재 대기 중인 VU 수)
- **Rate:** 백분율 (예: 성공률)
- **Trend:** 통계 데이터 (예: 응답 시간의 p95, p99)

### 3.4 Check & Threshold
- **Check:** 응답 조건 확인 (단순 검증, 테스트 중단 안 함).
- **Threshold:** 테스트 합격/불합격 기준 정의 (예: "95%의 응답 시간이 500ms 미만이어야 함").

## 4. 성능 측정 방법론

### 4.1 핵심 지표 (Key Metrics)
- **Throughput (TPS):** 시스템이 단위 시간당 처리하는 일량.
- **Latency (응답 시간):** 요청부터 응답까지 걸리는 시간.
  - **p50 (Median):** 가장 일반적인 사용자의 경험.
  - **p95/p99 (Tail Latency):** 하위 5%/1% 사용자가 겪는 지연. 동시성 경합 상황에서는 이 지표가 급증합니다.
- **Success Rate:** 전체 요청 중 비즈니스 로직 성공 비율.

### 4.2 부하 테스트의 종류
1. **Load Test:** 예상 트래픽 환경에서 시스템의 거동 확인.
2. **Stress Test:** 시스템이 중단되는 임계점 확인.
3. **Spike Test:** 갑작스러운 트래픽 급증 상황 시뮬레이션 (재고 선착순 이벤트).
4. **Soak Test:** 장시간 실행 시 메모리 누수나 자원 고갈 확인.

### 4.3 공정한 비교를 위한 환경 설정 (PoC 핵심)
본 PoC에서는 4가지 방식의 **순수 성능(Pure Performance)**을 비교하기 위해 다음을 준수합니다:
- **동일 리소스 제한:** Docker Compose의 `deploy.resources`를 통해 CPU/Memory 동일 할당.
- **Warm-up 단계:** JVM JIT 컴파일 및 캐시 활성화를 위한 사전 부하 수행.
- **데이터 격리:** 매 테스트마다 재고 수량과 Redis 상태 초기화 (`make reset`).
- **네트워크 노이즈 최소화:** 가급적 로컬 Docker 네트워크 내에서 수행.

## 5. 결과 분석 및 해석 방법론

### 5.1 상관관계 분석
- **TPS가 높고 Latency가 낮은가?** 최상의 시나리오.
- **TPS는 정체되는데 Latency만 늘어나는가?** 리소스 병목(Lock 경합) 발생 신호.
- **Latency는 낮은데 Success Rate가 낮은가?** 데이터 정합성 실패(Optimistic Lock 등).

### 5.2 동시성 제어 방식별 관전 포인트
- **Pessimistic Lock:** DB Lock 대기로 인한 Latency 증가 및 TPS 저하 확인.
- **Optimistic Lock:** 충돌 시 재시도(Retry)로 인한 오버헤드와 성공률 확인.
- **Redis Lock:** 네트워크 왕복 시간(RTT)이 성능에 미치는 영향 분석.
- **Lua Script:** 원자적 처리로 인한 가장 높은 TPS와 낮은 Latency 검증.
