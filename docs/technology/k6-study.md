# k6 Study: 부하 테스트 핵심 개념과 전략

본 문서는 동시성 제어 PoC의 성능 검증 도구로 사용된 **k6**의 기본 개념부터 심화 기능(Executor)까지 체계적으로 정리한 기술 문서입니다.

---

## 1. k6 개요 (Overview)

### 1.1 k6란?
Grafana Labs에서 개발한 오픈 소스 부하 테스트 도구로, **"코드로 정의하는 성능 테스트(Load Testing as Code)"**를 지향합니다.
Go 언어 기반의 고성능 엔진 위에서 JavaScript(ES6)로 시나리오를 작성할 수 있어 개발자 친화적입니다.

### 1.2 왜 k6인가?
- **Scriptable:** UI가 아닌 코드로 제어하므로 버전 관리(Git)와 재사용이 용이합니다.
- **Performance:** 단일 인스턴스에서 수천 명의 가상 사용자(VU)를 효율적으로 시뮬레이션합니다.
- **CI/CD Friendly:** CLI 기반으로 파이프라인 통합이 쉽습니다.

---

## 2. 핵심 용어 (Core Concepts)

### 2.1 VU (Virtual User)
- **가상 사용자**를 의미합니다.
- 각 VU는 독립적인 실행 루프(Loop)를 돌며 시나리오 코드를 병렬로 수행합니다.
- 운영체제 스레드와 유사하게 동작하지만 훨씬 가볍습니다.

### 2.2 Metrics (측정 지표)
- **RPS (Requests Per Second):** 초당 서버로 전송된 HTTP 요청 수.
- **TPS (Transactions Per Second):** 초당 완료된 비즈니스 트랜잭션(Iteration) 수.
- **Latency (지연 시간):**
    - `http_req_duration`: 요청 전송부터 응답 수신까지 걸린 시간.
    - `p95`: 하위 95% 요청의 최대 응답 시간 (Tail Latency).

### 2.3 Executor (실행기)
- VU를 어떻게 생성하고 스케줄링할지 결정하는 엔진입니다.
- *"100명이 10번씩 해라"* vs *"초당 100명이 들어오게 해라"* 처럼 부하의 성격을 정의합니다.

## 3. 심화 학습: Executor 3대장 (Deep Dive)

이번 프로젝트에서 사용된 3가지 핵심 Executor 패턴을 분석합니다.

### ① Shared Iterations (공유 반복) - "작업량 중심"
> **목적:** **Capacity Test** (최대 처리량 측정)

- **동작:** VU들이 하나의 작업 큐(총 Iterations)를 공유하여 처리합니다.
- **비유:** "100명의 일꾼이 벽돌 10,000장을 다 나를 때까지 일해라."
- **특징:** 시스템 성능이 좋을수록 테스트가 빨리 끝납니다. (시간 = 작업량 / TPS)
- **설정 예시:**
  ```javascript
  executor: 'shared-iterations',
  vus: 100,
  iterations: 10000, // 총 10,000번 실행하면 종료
  ```

### ② Constant VUs (고정 동시 접속) - "동시성 중심"
> **목적:** **Contention Test** (경합 및 안정성 검증)

- **동작:** 지정된 시간 동안 VU 수를 일정하게 유지합니다.
- **비유:** "5,000명이 30초 동안 쉬지 않고 문을 두드려라."
- **특징:** 서버가 죽거나 응답이 늦어져도 사용자 수는 유지되므로, 극한의 경합 상황을 연출하기 좋습니다.
- **설정 예시:**
  ```javascript
  executor: 'constant-vus',
  vus: 5000,
  duration: '30s', // 30초간 5,000명 유지
  ```

### ③ Ramping Arrival Rate (단계적 부하) - "임계점 탐색"
> **목적:** **Stress Test** (Knee Point 탐색)

- **동작:** VU 수가 아닌 **RPS(초당 요청 수)**를 목표로 삼고, 이를 달성하기 위해 VU를 동적으로 조절합니다.
- **비유:** "손님을 초당 10명에서 시작해서, 점차 늘려 초당 1,000명까지 입장시켜 봐."
- **특징:** 시스템이 언제 뻗는지(Latency가 튀는 지점)를 정확히 찾을 수 있습니다.
- **설정 예시:**
  ```javascript
  executor: 'ramping-arrival-rate',
  stages: [
      { duration: '1m', target: 500 },  // 1분 동안 500 RPS까지 증가
      { duration: '3m', target: 2000 }, // 그 후 3분 동안 2,000 RPS까지 증가
  ],
  ```

## 4. 프로젝트 적용 전략 (Best Practices)

본 PoC에서는 정확한 성능 측정을 위해 다음과 같은 원칙을 적용했습니다.

### 4.1 격리성 (Isolation)
- **문제:** 이전 테스트의 커넥션 풀이나 캐시가 남아있으면 다음 테스트 결과가 왜곡됩니다.
- **해결:** `make clean && make up` 명령어로 매 테스트마다 인프라를 완전히 재시작합니다.

### 4.2 Warm-up (예열)
- **문제:** Java 애플리케이션은 JIT 컴파일과 커넥션 풀 초기화 때문에 처음 몇 초간 성능이 낮습니다.
- **해결:** 본 테스트 직전에 가벼운 부하(10 VUs)를 10초간 주어 시스템을 예열합니다.

### 4.3 Host Proxy Bypass
- **문제:** 로컬(Host)에서 `k6`를 실행하면 Docker 네트워크로 진입할 때 프록시 오버헤드가 발생할 수 있습니다.
- **해결:** `k6` 자체를 Docker 컨테이너로 띄워 내부망(`concurrency-network`)에서 직접 통신하게 하여 네트워크 병목을 제거했습니다.

## 5. 결론

k6는 단순한 트래픽 생성기가 아니라, 목적에 따라 정교하게 부하를 설계할 수 있는 강력한 도구입니다.
이번 프로젝트에서는 **Executor**의 특성을 활용하여 **Capacity, Contention, Stress**라는 3가지 차원에서 시스템 성능을 입체적으로 검증해냈습니다.
