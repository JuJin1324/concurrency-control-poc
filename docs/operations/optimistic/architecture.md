# [Deep Dive] Optimistic Lock Architecture Patterns

**Parent Document:** [Optimistic Lock 운영 가이드](../optimistic-lock-ops.md)

낙관적 락은 "충돌은 드물다"는 가정하에 출발하지만, 실제 대규모 환경에서는 그 가정이 깨질 때 발생하는 **재시도 폭풍(Retry Storm)**이 가장 큰 아키텍처적 난제입니다. 이를 극복한 글로벌 기업들의 사례를 분석합니다.

---

## 1. 숙박 예약: Airbnb (The Lesson of High Contention)
**핵심 교훈:** 고경합(High Contention) 상황에서 순수 낙관적 락은 시스템을 파괴한다.

### 1.1 초기 문제: 재시도 폭풍 (Retry Storm)
*   **상황:** 인기 숙소 예약 시 수천 명이 동시에 결제 시도.
*   **동작:**
    1.  1,000명이 읽음 (Version 1).
    2.  1명이 수정 성공 (Version 2).
    3.  999명이 충돌 발생(`OptimisticLockException`) -> **전원 즉시 재시도**.
    4.  999명 중 1명 성공 -> 998명 재시도...
*   **결과:** DB CPU가 기하급수적으로 상승하고, 성공해야 할 요청조차 타임아웃으로 실패. (이론적으로 비용은 $O(N^2)$로 증가)

### 1.2 해결책: Soft Holds (방파제 전략)
*   **Redis 도입:** DB 낙관적 락으로 가기 전에, Redis에서 먼저 **임시 점유(Soft Hold)**를 획득하게 함.
*   **동작:** `SET resource:id user:id NX EX 300` (300초 점유)
*   **효과:** Redis 락을 획득한 소수만 DB 트랜잭션에 진입하므로, DB 레벨의 충돌 비용을 획기적으로 낮춤. (하이브리드 전략)

## 2. 실시간 협업: Notion (Granularity Strategy)
**핵심 교훈:** 락의 범위(Granularity)를 쪼개면 충돌 확률이 0에 수렴한다. 기술적 락킹보다 데이터 모델링이 중요하다.

### 2.1 문제 정의: 페이지 단위 락킹의 한계
*   **구조:** `Page` 테이블에 `content` 컬럼(JSON)이 있는 형태라면?
*   **문제:** 사용자 A는 상단 제목을 고치고, B는 하단 표를 고침. 서로 다른 영역이지만 `Page`의 `version`이 하나뿐이라 무조건 충돌 발생. 협업 불가능.

### 2.2 해결책: Block-level Locking (데이터 모델링의 승리)
*   **전략:** Notion은 페이지를 텍스트 덩어리로 저장하지 않고, 수천 개의 독립적인 **'블록(Block)'** 엔티티로 쪼개서 관리함.
*   **구조:**
    *   `Page(id, title, version)`
    *   `Block(id, page_id, content, version)` -> **각 블록이 독립적인 버전을 가짐.**
*   **효과:**
    *   A는 `Block 1`의 버전을 올림 (v1 -> v2).
    *   B는 `Block 99`의 버전을 올림 (v5 -> v6).
    *   **결과:** 서로 다른 DB Row를 수정하므로 **물리적 충돌이 전혀 발생하지 않음.** 낙관적 락을 쓰면서도 실시간 동시 편집을 완벽하게 지원.

## 3. 위키 도구: Confluence (Conflict Resolution Strategy)
**핵심 교훈:** 충돌을 피할 수 없다면, **"자동 병합(Auto-Merge)"**과 **"사용자 개입(Manual Resolution)"**으로 해결한다.

### 3.1 문제 정의
*   Notion처럼 블록 단위가 아닌 '페이지 단위' 락킹을 사용하는 레거시 구조나 문서 형태인 경우, 사용자 A와 B가 동시에 편집하면 반드시 충돌이 발생.

### 3.2 해결책: 3-Way Merge & UX
*   **Auto-Merge (자동 병합):**
    *   사용자가 저장을 시도할 때, 서버는 `Base Version`(사용자가 읽은 시점)과 `Current Version`(DB 최신)을 비교.
    *   **로직:** 두 버전 간의 Diff(변경분)를 계산하여, 서로 다른 섹션(문단)을 수정했다면 서버가 **자동으로 병합**하여 저장. 사용자에게는 "저장됨"만 보여줌.
*   **Conflict Wizard (수동 해결):**
    *   같은 문단을 수정하여 자동 병합이 불가능한 경우(Overlap), 409 에러만 뱉지 않고 **충돌 해결 마법사** UI를 띄움.
    *   **옵션:**
        1.  **Overwrite:** "내 변경사항으로 덮어쓰기" (상대방 작업 무시)
        2.  **Re-edit:** "상대방 변경사항 불러오기" (내 작업 임시 저장 후 다시 편집)
        3.  **Cancel:** "취소"

## 4. E-commerce: Salesforce Commerce Cloud
**핵심 교훈:** API 레벨에서의 명확한 프로토콜 정의와 운영 효율성 확보.

### 4.1 도메인 및 선택 이유
*   **도메인:** 전자상거래 플랫폼 (재고 관리, 상품 정보).
*   **특성:** 읽기 작업(Read)이 쓰기(Write)보다 압도적으로 많음. 실제 가격/재고 수정은 소수 관리자에 의해 수행됨.
*   **이유:** 초기에는 Pessimistic Lock을 사용했으나 플랫폼 성장에 따라 성능 병목이 발생하여 Optimistic Lock으로 전환.

### 4.2 구현 방식: OCAPI (Open Commerce API)
*   **Version Column:** 모든 리소스에 버전을 부여하여 충돌 감지.
*   **HTTP 409 Conflict:** 충돌 발생 시 서버는 `500 Internal Server Error`가 아닌, 명확한 `409 Conflict` 상태 코드와 `ConcurrentModificationException`을 반환.

### 4.3 운영 노하우 및 경험
*   **충돌률 데이터:** 공식 문서에 따르면 충돌은 대부분 **"Bad Timing"**으로 인해 드물게 발생하며, "간단한 재시도로 대부분 해결 가능"하다고 명시.
*   **Retry 전략:**
    *   **클라이언트 주도 재시도(Client-side Retry):** 서버가 자동으로 재시도하지 않고 에러를 반환.
    *   **Simple Retry:** 복잡한 Backoff 없이 단순 재시도만으로도 성공률이 높음 (경합이 낮기 때문).
*   **모니터링:** HTTP 409 응답 발생 빈도를 추적하여, 평소보다 급증하면 시스템 부하 이슈로 판단.
*   **성과:** Pessimistic Lock 대비 시스템 전체의 **처리량(Throughput)**이 비약적으로 상승하고 운영이 원활해짐.

## 5. 플랫폼: GitHub (Online Schema Migration)
**핵심 교훈:** 서비스 중단 없는 대규모 데이터 이동의 핵심 기술. `updated_at`을 버전으로 활용한 낙관적 검증.

### 5.1 문제: 달리는 자동차의 바퀴 갈아 끼우기 (Downtime)
*   **상황:** 10억 건의 데이터가 있는 `User` 테이블에 컬럼을 추가하거나 데이터 타입을 변경해야 함.
*   **전통적 방식:** `ALTER TABLE` 명령은 테이블 전체에 락(Table Lock)을 걸어 수 시간 동안 서비스 중단을 야기함.

### 5.2 해결책: gh-ost(GitHub Online Schema Transmogrifier) 아키텍처
GitHub는 테이블을 직접 수정하지 않고, **낙관적 검증을 통한 데이터 복제** 방식을 사용함.

1.  **Ghost Table 생성:** 원본과 동일하지만 스키마가 변경된 새로운 테이블(`User_New`)을 생성.
2.  **Chunk 복사:** 원본 테이블에서 데이터를 일정 범위(예: 1,000건)씩 쪼개어 새 테이블로 복사. 이 과정은 락 없이 수행됨.
3.  **Binlog 스트리밍:** 복사가 진행되는 동안 유저가 발생시킨 변경 사항(INSERT/UPDATE/DELETE)을 별도의 로그(Binlog)로 실시간 수집.
4.  **Optimistic Verification (Copy-and-Verify):**
    *   **Version 식별:** 원본 로우의 `updated_at` 타임스탬프를 **버전(Version)**으로 간주.
    *   **검증:** 데이터를 복사하여 `User_New`에 넣기 직전, 다시 한번 원본의 `updated_at`을 확인.
    *   **조치:** "내가 읽어온 시점보다 `updated_at`이 더 최신이라면?", 그 사이 유저가 데이터를 수정한 것이므로 복사를 중단하고 최신 데이터를 다시 읽어와서 복제함.
5.  **결과:** 단 1초의 서비스 중단 없이 10억 건의 데이터 이동과 스키마 변경을 완벽하게 완료.

## 6. 예약 플랫폼: Booking.com (Read-Optimized Hybrid)
**핵심 교훈:** 읽기는 관대하게(Optimistic), 쓰기는 엄격하게(Pessimistic). 비즈니스 성공을 위한 기술적 타협.

### 6.1 전략: 낙관적 읽기 + 비관적 쓰기
Booking.com은 호텔 예약이라는 도메인 특성상, "예약 가능"이라고 표시된 방이 실제 결제 시점에 "예약 불가"로 바뀌는 UX(Negative UX)가 매출에 치명적입니다. 하지만 모든 조회에 락을 걸면 시스템이 죽습니다.

#### Phase 1: 검색 (Read Optimistic)
*   **동작:** 사용자가 호텔을 검색하고 상세 페이지를 볼 때는 **어떤 락도 걸지 않습니다(No Lock).**
*   **특징:**
    *   심지어 Slave DB나 캐시(Redis)에서 5초 전 데이터를 보여줄 수도 있습니다.
    *   **의도:** 0.1초 전에는 방이 있었지만, 지금은 나갔을 수도 있습니다. 하지만 이를 허용합니다. 조회 트래픽이 예약 트래픽보다 100배 이상 많기 때문에, 여기서 정합성을 따지면 서비스 운영이 불가능합니다.
    *   **심리적 장치:** "지금 3명이 이 방을 보고 있습니다" 같은 문구로 긴장감을 주어 빠른 결제를 유도합니다.

#### Phase 2: 결제 진입 (Write Pessimistic)
*   **동작:** 사용자가 "예약하기" 버튼을 누르는 순간, 시스템은 해당 객실 재고에 대해 강력한 **비관적 락(Pessimistic Lock)**이나 Redis 분산 락을 획득합니다.
*   **이유:**
    *   결제 단계에 진입한 고객은 **구매 전환율(Conversion Rate)**이 매우 높은 진성 고객입니다.
    *   이 단계에서 "재고가 변경되었습니다"라며 튕겨내면 고객은 경쟁사로 이탈합니다.
    *   따라서 트래픽 처리 비용을 더 지불하더라도, 이 단계에서는 **확실한 선점**을 보장하는 것이 비즈니스적으로 이득입니다.

## 7. 부모-자식 관계의 락킹 전략 (Aggregate Locking)
자식 엔티티(Child)가 수정될 때 부모 엔티티(Parent)의 버전을 올릴 것인가? 이는 **동시성**과 **정합성** 사이의 트레이드오프입니다.

### 7.1 전파 전략 (Propagation - DDD Style)
*   **동작:** 자식(`OrderItem`)이 추가/수정되면 부모(`Order`)의 버전도 강제로 증가시킴. (`OPTIMISTIC_FORCE_INCREMENT`)
*   **이유:** 자식의 변경이 부모의 불변식(Invariant, 예: 주문 총액)에 영향을 줄 때. 애그리거트 전체의 일관성을 강력하게 보장.
*   **단점:** 자식 A를 수정하는 동안 자식 B를 수정하려는 트랜잭션도 충돌 발생 (동시성 저하).

### 7.2 독립 전략 (Independence - High Concurrency)
*   **동작:** 자식(`Comment`)이 추가되어도 부모(`Post`)의 버전은 그대로 둠.
*   **이유:** 자식의 변경이 부모의 상태와 무관할 때. (예: 게시글 내용과 댓글은 별개)
*   **장점:** 부모 락 경합 없이 여러 사용자가 동시에 댓글을 달 수 있음. (Notion의 Block 방식과 유사)
