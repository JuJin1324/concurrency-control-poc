# [Deep Dive] Optimistic Lock Troubleshooting & Checklists

**Parent Document:** [Optimistic Lock 운영 가이드](../optimistic-lock-ops.md)

낙관적 락 운영의 핵심은 **"충돌(Conflict)을 어떻게 관리할 것인가"**입니다. 충돌은 에러가 아니라 **예상된 비즈니스 흐름**으로 취급되어야 합니다. 이 문서는 재시도 폭풍(Retry Storm)의 원리를 수학적으로 분석하고, 이를 방어하기 위한 고급 전략을 제시합니다.

---

## 1. 재시도 폭풍 (Retry Storm): 수학적 공포

낙관적 락이 실패하는 가장 큰 이유는 **경쟁 심화(High Contention)** 상황에서의 비효율성입니다.

### 1.1 충돌 비용의 기하급수적 증가 ($O(N^2)$)
100명의 사용자가 동시에 하나의 자원(`version=1`)을 수정하려고 시도한다고 가정해봅시다.

*   **Round 1:** 100명 시도 -> 1명 성공(v2), 99명 실패. (총 100 트랜잭션)
*   **Round 2:** 실패한 99명 재시도 -> 1명 성공(v3), 98명 실패. (총 99 트랜잭션)
*   **Round 3:** 실패한 98명 재시도 -> 1명 성공(v4), 97명 실패. (총 98 트랜잭션)
*   ...
*   **총 트랜잭션 수:** $100 + 99 + 98 + ... + 1 \approx 5,050$회.

겨우 100개의 요청을 처리하기 위해 DB는 **5,000번 이상의 트랜잭션**을 감당해야 합니다. 이것이 바로 DB CPU를 100%로 만들고 시스템을 마비시키는 **재시도 폭풍**의 실체입니다.

---

## 2. 재시도 전략 (Retry Strategy): "과학적인 후퇴"

단순 `while(true)` 루프는 시스템을 자가 파괴(Self-destruction)하게 만듭니다. 우리는 **Backoff**와 **Jitter**로 대응해야 합니다.

### 2.1 Exponential Backoff (지수 백오프)
실패 횟수가 늘어날수록 대기 시간을 배수로 늘려 DB에 숨 쉴 틈을 줍니다.
*   `wait_time = base * 2^attempt`
*   예: 100ms -> 200ms -> 400ms -> 800ms ...

### 2.2 Jitter (지터): 무작위성의 마법
지수 백오프만으로는 부족합니다. 모든 클라이언트가 똑같이 100ms 뒤에 깨어나면, 그 순간 다시 충돌합니다. 이를 **Lockstep(집단 동조) 현상**이라고 합니다. 이를 깨기 위해 **무작위성(Randomness)**을 부여합니다.

#### AWS 권장 알고리즘: Full Jitter
가장 효율적으로 부하를 분산시키는 알고리즘입니다.
*   **공식:** `wait_time = random_between(0, min(cap, base * 2^attempt))`
*   **동작:** 단순히 고정된 시간을 기다리는 게 아니라, **0부터 지수적으로 증가한 최대 시간 사이의 랜덤한 값**을 선택합니다.
*   **효과:** 어떤 클라이언트는 10ms 뒤에, 어떤 클라이언트는 300ms 뒤에 재시도하게 되어 요청이 시간 축상에 고르게 퍼집니다(Spread out).

## 3. 모니터링 및 임계값 (Metrics & Thresholds)

낙관적 락은 **성공보다 실패(충돌) 정보가 더 중요**합니다. 다음 지표를 반드시 모니터링해야 합니다.

### 3.1 핵심 지표
*   **`optimistic_lock_failure_rate`:** 전체 요청 대비 충돌 비율.
*   **`average_retry_count`:** 성공한 요청들이 평균적으로 몇 번 재시도했는가?

### 3.2 위험 신호 (Alert Thresholds)
1.  **Failure Rate > 5%:** 전체 트랜잭션의 5% 이상이 충돌로 실패한다면, 낙관적 락을 쓰기에 부적절한(너무 뜨거운) 리소스입니다. 비관적 락이나 큐 도입을 고려해야 하는 **전환점(Pivot Point)**입니다.
2.  **Avg Retry Count > 1.5회:** 대부분의 요청이 한 번에 성공하지 못하고 평균 1.5번 이상 재시도하고 있다면, 이미 DB 리소스의 50% 이상이 낭비되고 있다는 뜻입니다.

## 4. 출구 전략 (Exit Strategy): 언제 포기할 것인가?

무한 재시도는 시스템을 좀비로 만듭니다. 명확한 포기 기준이 필요합니다.

*   **전략 A: 사용자에게 위임 (Fail-Fast)**
    *   최대 3~5회 재시도 후에도 실패하면 `MaxRetryExceededException`을 던집니다.
    *   "현재 사용자가 많아 처리가 지연되고 있습니다. 잠시 후 다시 시도해주세요."
*   **전략 B: 비관적 락으로 전환 (Dynamic Pivot)**
    *   특정 리소스에서 연속으로 충돌이 발생하면, 마지막 시도에서는 `PESSIMISTIC_WRITE` 락을 걸고 확실하게 처리하는 하이브리드 전략입니다. (구현 복잡도 높음)

## 5. UX 처리 방안 (Conflict Psychology)

기술적 재시도만으로는 해결되지 않는 "논리적 충돌"은 사용자에게 선택권을 줘야 합니다.

### 5.1 Silent Retry (투명한 처리)
*   **대상:** 내부 카운터, 로그성 데이터.
*   **동작:** 사용자는 모르게 서버가 알아서 재시도하여 성공시킴.

### 5.2 Auto-Merge (자동 병합)
*   **대상:** Notion/Wiki의 서로 다른 블록 수정.
*   **동작:** 최신 버전을 다시 읽어와서(`re-read`), 내 변경 사항(Delta)만 다시 적용(`re-apply`). 충돌이 없으면 저장.

### 5.3 User Intervention (사용자 개입)
*   **대상:** 티켓팅, 동일 텍스트 동시 수정.
*   **동작:** "죄송합니다. 다른 사용자가 먼저 예약했습니다." 팝업 노출. 사용자가 포기하거나 다른 좌석을 고르게 유도.
