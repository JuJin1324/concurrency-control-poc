# [Deep Dive] Pessimistic Lock DB Internals

**Parent Document:** [Pessimistic Lock 운영 가이드](../pessimistic-lock-ops.md)

비관적 락은 데이터베이스 엔진의 가장 깊숙한 곳에서 동작하는 기능입니다. 이 문서는 MySQL(InnoDB)와 PostgreSQL이 어떻게 락을 처리하는지, 그리고 MVCC와 락이 어떻게 공존하는지에 대한 기술적 심층 분석을 제공합니다.

---

## 1. MVCC와 Lock의 공존 철학: "왜 둘 다 필요한가?"

많은 개발자들이 **"MVCC(Multi-Version Concurrency Control)가 있는데 왜 락을 써야 하죠?"**라고 묻습니다. 이 둘은 대체제가 아니라 상호보완재입니다.

### 1.1 MVCC = RDBMS 내의 Eventual Consistency
*   **본질:** "읽기(Reader)"와 "쓰기(Writer)"가 서로를 차단하지 않게 하는 평화 협정입니다.
*   **동작 원리:**
    *   트랜잭션 A가 데이터를 수정 중일 때, 트랜잭션 B가 조회를 요청하면 락을 대기하는 대신 **Undo Log에 있는 과거 버전(Snapshot)**을 읽습니다.
    *   **의미:** "지금 당장 최신(`Current`) 데이터가 아닐 수도 있지만, 읽는 데 방해는 안 받게 해줄게. 너만의 시간선에서 일관성 있게 읽어."
*   **한계 (갱신 손실):** 데이터를 읽어서 수정한 뒤 다시 저장하는 **Read-Modify-Write** 패턴에서는 치명적입니다. 과거 데이터를 보고 수정하면, 다른 트랜잭션의 수정 사항을 덮어써버리기 때문입니다.

### 1.2 비관적 락 = Strong Consistency + Serialization
*   **본질:** MVCC의 "안 기다리기" 특성을 포기하고, **"줄 서기(Queueing)"**를 강제하는 행위입니다.
*   **동작 원리:**
    *   `SELECT ... FOR UPDATE`를 실행하면, DB는 스냅샷이 아니라 **무조건 현재 시점의 최신 값(Current Read)**을 가져옵니다.
    *   그리고 트랜잭션이 끝날 때까지 해당 데이터에 **배타 락(X-Lock)**을 걸어 다른 누구도 건드리지 못하게 합니다.
*   **대가:** 동시성을 희생하여 데이터 정합성을 삽니다. (Serialization)

### 1.3 언제 락을 써야 하는가? (Decision Matrix)

| 상황 | 전략 | 이유 |
| :--- | :--- | :--- |
| **단순 조회** (상품 상세, 게시글) | **MVCC (No Lock)** | 최신성이 0.1초 늦어도 됨. 읽기가 쓰기를 방해하면 안 됨. |
| **통계/리포트** (집계) | **MVCC (Snapshot Read)** | 락을 걸면 서비스가 멈춤. 일관된 과거 시점 데이터면 충분함. |
| **개인 정보 수정** (프로필) | **MVCC (+ 낙관적 락)** | 나만 쓰는 데이터임. 동시 수정(PC/모바일) 충돌만 감지하면 충분함. 굳이 미리 잠글 필요 없음. |
| **재고/잔액 차감** (공유 자원 수정) | **비관적 락** | "남은 수량"을 확인하고 깎아야 함. 타인과의 동시 수정을 직렬화해야 함. |
| **선착순/상태 변경** | **비관적 락** | "내가 찜했다"는 사실을 물리적으로 확정해야 함. |
| **데이터 정합성 체크** (FK 확인) | **비관적 락 (`FOR SHARE`)** | "내가 자식 데이터를 넣을 동안 부모 너 삭제되지 마" (읽기지만 상태 고정 필요) |

## 2. MySQL InnoDB: 인덱스 기반 락킹의 함정

MySQL InnoDB 엔진에서 가장 빈번하게 발생하는 오해는 락이 '물리적인 레코드(Row)' 자체에 걸린다는 생각입니다. 하지만 InnoDB는 레코드가 아닌 **인덱스(Index)**에 락을 겁니다.

### 2.1 Clustered Index (PK) Lock
*   **동작:** `WHERE id = 1 FOR UPDATE` (PK 사용)
*   **효과:** Primary Key 인덱스의 해당 레코드 하나만 정확히 잠급니다. (Record Lock)
*   **권장:** 가장 이상적인 형태이며 경합을 최소화합니다.

### 2.2 Gap Lock & Next-Key Lock (재앙의 시작)
*   **상황:** 범위 검색이나 Non-Unique 인덱스를 사용할 때.
*   **동작:** `SELECT * FROM orders WHERE status = 'PENDING' FOR UPDATE`
*   **메커니즘:**
    *   InnoDB는 **Phantom Read** (트랜잭션 도중 없던 데이터가 생기는 현상)를 방지하기 위해 락을 겁니다.
    *   현재 `PENDING`인 주문뿐만 아니라, **앞으로 들어올(INSERT 될) `PENDING` 주문의 자리(Gap)**까지 잠가버립니다.
*   **위험:** 신규 주문 생성(`INSERT`) 트랜잭션까지 대기 상태에 빠져 전체 시스템의 처리량이 급감합니다.

### 2.3 해결책: 2-Phase Locking
Gap Lock을 피하기 위해 다음과 같은 패턴을 사용합니다.
1.  **Step 1 (No Lock):** 먼저 락 없이 조건에 맞는 ID만 조회합니다. (`SELECT id ...`)
2.  **Step 2 (Lock by PK):** 조회된 ID로 정확하게 찍어서 락을 겁니다. (`WHERE id IN (...) FOR UPDATE`)

## 3. PostgreSQL: 세분화된 락 모드의 활용

PostgreSQL은 MySQL보다 훨씬 다층적인 락 모드를 제공하여, 개발자가 비즈니스 로직의 특성에 맞춰 동시성을 미세 조정할 수 있게 합니다.

### 3.1 `FOR UPDATE`
*   **설명:** 가장 강력한 락. 해당 행의 삭제(`DELETE`) 및 모든 컬럼의 업데이트를 차단합니다.
*   **용도:** 잔액 차감, 재고 차감 등 핵심 데이터 수정 시.

### 3.2 `FOR NO KEY UPDATE` (Best Practice 🌟)
*   **설명:** "키(PK, UK) 컬럼을 제외한 나머지 컬럼의 업데이트는 막지만, **PK를 건드리지 않는다면 다른 작업(참조 등)은 허용**해 줄게."
*   **용도:** 주문 상태 변경 (`PENDING` -> `PAID`).
*   **장점:**
    *   부모 테이블(주문)을 잠가도, 자식 테이블(주문 상세)에 데이터를 추가(`INSERT`)하는 트랜잭션이 블로킹되지 않습니다.
    *   부모의 PK가 바뀌지 않는다는 것만 보장되면, 자식이 참조하는 데는 문제가 없기 때문입니다.
    *   **동시성을 획기적으로 높일 수 있는 PostgreSQL만의 강점입니다.**

### 3.3 `FOR SHARE`
*   **설명:** "나 이거 읽고 있으니까, **수정하거나 삭제하지 마.** (읽는 건 같이 읽자)"
*   **용도:** 리포트 생성, 정산 집계. 데이터 불변성이 필요할 때. (낙관적 락과는 다릅니다. 수정을 적극적으로 차단합니다.)

### 3.4 `FOR KEY SHARE`
*   **설명:** 가장 약한 락. "이 데이터가 **삭제되지만 않으면 돼.** (PK만 유지되면 됨)"
*   **용도:** 외래 키(FK) 정합성 체크 시 DB가 내부적으로 사용합니다. 개발자가 직접 호출할 일은 적지만, FK로 인한 데드락 원인을 파악할 때 알아야 합니다.
