# [Deep Dive] Redis Distributed Lock Architecture Patterns

**Parent Document:** [Redis Distributed Lock 운영 가이드](../redis-lock-ops.md)

Redis 분산 락은 분산 환경에서 상호 배제(Mutual Exclusion)를 구현하기 위한 사실상의 표준입니다. 이 문서는 국내외 선도 기업들의 실제 도입 사례와 장애 경험, 그리고 인프라 구성에 따른 트레이드오프를 심층 분석합니다.

---

## 1. 엔터프라이즈 도입 사례 분석 (Case Studies)

### 1.1 우아한형제들 (배달의민족): 초고부하 선착순 시스템
**전략:** "락을 걸지 않는 것이 가장 빠른 락이다" (Redis Set 활용 & Redisson)

*   **문제 정의:**
    *   선착순 쿠폰 이벤트 시 수만 명의 트래픽이 몰림.
    *   DB 락(`SELECT FOR UPDATE`)을 쓰면 커넥션 풀이 말라버리고, 일반 Redis 락(`LOCK` -> `CHECK` -> `UNLOCK`)을 써도 네트워크 오버헤드가 큼.
*   **해결책 1: Lock-free Pattern (Set 자료구조)**
    *   **비유:** 문 앞에 줄을 세우고 한 명씩 들여보내는(Lock) 대신, **"방명록(Set)에 이름부터 쓰세요"**라고 하는 방식.
    *   **동작:** `SADD event:users {userId}` 명령어는 유저가 처음이면 `1`, 이미 있으면 `0`을 반환함. 이 반환값만 확인하면 락 없이도 중복을 100% 막을 수 있음.
        *   **수량 제어:** `SCARD coupon:issued:users`로 현재 발급량을 조회하여 제한 수량 확인.
        *   **핵심:** 명시적인 `tryLock` / `unlock` 과정 없이, Redis 자료구조 자체의 특성을 이용해 동시성을 제어함으로써 오버헤드를 0으로 만듦.
        *   **Why Common?:** 대부분의 선착순 이벤트는 **"1인 1회"** 제한이 걸려있습니다. `SADD`는 중복 방지(Idempotency)가 내장되어 있어 별도 로직 없이도 "따닥"을 원천 차단하므로 실무에서 표준처럼 쓰입니다.
        *   **Trade-off:** 만약 **"1인 4매 티켓팅"**처럼 수량 제한이 있는 경우라면 `SADD`만으로는 부족합니다. 이때는 `Hash`(`HINCRBY`) 구조를 사용하여 유저별 구매 수량을 별도로 카운팅해야 합니다.
    *   **Redisson 도입:** 주문 데이터 생성 등 락이 필수적인 로직에는 Redisson의 **Watchdog** 기능을 활용하여 비즈니스 로직 수행 중 락이 만료되는 것을 방지.
    *   초기에는 `Lettuce`로 스핀 락을 구현했으나, 락을 못 얻은 대기열 트래픽이 Redis에 계속 "자리가 났나요?"라고 물어보느라(Spinning) Redis CPU가 100%를 치는 장애 발생.
    *   **Redisson**으로 교체하여, Pub/Sub 방식으로 "자리가 나면 알려주는" 구조로 변경해 CPU 부하를 해소함.

### 1.2 토스 (Toss): 결제 정합성과 하이브리드 락킹
**전략:** 개발자 실수를 막는 래퍼(Wrapper)와 2중 방어선

*   **문제 정의:**
    *   결제 시스템에서 '따닥(중복 요청)'은 절대 허용될 수 없음.
    *   Redisson은 기능이 강력하지만 무거움. 가벼운 `Lettuce`를 쓰되 안전하게 쓰고 싶음.
*   **해결책 1: Safety Wrapper (Lettuce + Fail-Fast)**
    *   **Why Lettuce?:** Redisson은 무거운 의존성과 부가 기능(Watchdog 등)을 가짐. 토스는 이를 덜어내고 `SETNX`만 사용하는 경량 방식을 택함.
    *   **No Spin:** 스핀 락의 부하를 막기 위해, 락 획득 실패 시 대기하지 않고 즉시 **"잠시 후 다시 시도해주세요"** 에러를 반환(Fail-Fast)하는 UX 전략을 사용.
    *   **Wrapper:** Kotlin 고차 함수(`useLock`)로 감싸서 `finally { unlock }`을 강제함.
*   **해결책 2: Hybrid Defense (Redis + DB)**
    *   **1차 방어 (Redis):** 분산 락으로 99.9%의 트래픽 제어.
    *   **2차 방어 (DB Unique Key):** `PaymentHistory` 테이블에 `UNIQUE INDEX (order_id)`를 설정. Redis가 뚫려도 DB `INSERT` 시점에 `DuplicateKeyException`이 발생하여 물리적으로 중복 결제를 차단.
    *   **3차 방어 (Optimistic Lock):** 잔액 차감(`UPDATE`) 시에는 `@Version`으로 방어.

### 1.3 쿠팡 (Coupang): SPOF 장애의 교훈
**전략:** "실패를 0으로 덮지 마라" (Graceful Degradation)

*   **사건:** 2019년 모든 상품이 '품절'로 표시되어 주문 불가.
*   **원인 (The Butterfly Effect):**
    1.  재고 조회를 위해 Redis를 호출했으나 과부하로 **Timeout Exception** 발생.
    2.  개발자 코드는 예외 발생 시 안전하게 **`return 0` (재고 없음)**을 반환하도록 되어 있었음.
    3.  **결과:** Redis가 느려지자 모든 상품의 재고가 0으로 인식되어 전면 품절 사태 발생.
*   **교훈 및 해결책:**
    *   **예외 처리:** 장애를 비즈니스 로직(품절)으로 둔갑시키지 말고, 명확한 에러를 뱉거나 다른 경로를 찾아야 함.
    *   **Circuit Breaker & Fallback:** Redis 오류율이 임계치를 넘으면 서킷을 열고(Open), **Local Cache(Ehcache)**를 조회하거나 **DB를 직접 조회**하는 방식으로 우회하여 서비스 가용성을 유지함.

### 1.4 그랩 & 우버 (Grab/Uber): 워크플로우 엔진으로의 진화
**전략:** "시간에 의존하는 락(TTL)을 버리고, 상태(State)를 기록하라"

*   **문제: TTL의 딜레마 (Long-running Transaction)**
    *   택시 배차는 기사님이 수락할 때까지 10초~30초가 걸리는 긴 작업입니다.
    *   **TTL이 너무 짧으면:** 기사님이 고민하는 사이에 락이 풀려서 다른 승객이 배차됩니다 (중복 배차).
    *   **TTL이 너무 길으면:** 기사님 앱이 강제 종료되어도 락이 남아있어, 한동안 다른 콜을 못 받습니다 (가용성 저하).
    *   **결론:** 시간(Time)에 의존하는 Redis 락으로는 사람의 행동(Human Interaction)을 제어하기 어렵습니다.
*   **진화: 상태 기반 동시성 제어 (State-based Concurrency)**
    *   그랩은 Redis 락을 버리고 **Temporal(워크플로우 엔진)**을 도입했습니다.
    *   **방식:** "락"을 거는 게 아니라, DB에 **"현재 기사 B는 승객 A의 요청을 검토 중"**이라는 **상태(State)**를 영구적으로 기록합니다.
    *   **효과:** 서버가 재시작되거나 시간이 오래 지나도 상태는 사라지지 않으므로, 기사님의 응답(수락/거절)이 올 때까지 안정적으로 대기할 수 있습니다.

## 2. Redis 토폴로지별 락 안정성 분석

어떤 Redis 구성을 사용하느냐에 따라 락의 신뢰성 수준이 결정됩니다.

| 구성 방식 | 아키텍처 특징 | 분산 락 관점의 장점 | 위험 요소 및 제약 |
| :--- | :--- | :--- | :--- |
| **Standalone** | 단일 마스터 | 구현이 매우 간단함. 네트워크 홉이 적어 가장 빠름. | **SPOF**: 마스터 장애 시 락 전면 중단. |
| **Sentinel** | 마스터-슬레이브 + 자동 페일오버 | 자동 복구(HA) 확보. | **Split-Brain (뇌 분열):** 마스터가 죽고 슬레이브가 승격되는 찰나의 순간, 기존 마스터에 있던 락 정보가 복제되지 않아 **두 명이 동시에 락을 잡는 사고** 발생 가능. |
| **Cluster** | 데이터 샤딩 + 페일오버 | 대규모 트래픽 처리 및 수평 확장 용이. | Sentinel과 동일한 복제 지연 문제 내재. 락 키가 서로 다른 슬롯에 있으면 트랜잭션 처리가 까다로움. |

*   **배민/토스의 선택:** Sentinel (또는 Cluster) + **DB 2차 검증**.
    *   "Redis가 100% 완벽하지 않다(Split-Brain)"는 것을 인정하고, 아주 드물게 락이 깨지는 경우는 DB에서 막겠다는 실용적인 접근.

## 3. 네트워크 지연(RTT)과 성능

*   **영향:** 분산 락 성능은 Redis 자체 속도(CPU)보다 **클라이언트-Redis 간의 왕복 시간(RTT)**에 좌우됨. (왔다 갔다 하는 횟수가 많으므로)
*   **최적화:**
    *   **Locality:** Redis와 앱 서버를 **같은 가용 영역(AZ)**에 배치. (RTT 1ms 미만 필수)
    *   **Lua Script:** "락 있는지 확인하고, 없으면 걸고, TTL 설정해"라는 3번의 과정을 스크립트 하나로 묶어서 한 번만 왔다 갔다 하게 함.
