# [Deep Dive] Redis Lua Script Mechanics

**Parent Document:** [Lua Script 운영 가이드](../lua-script-ops.md)

이 문서는 Redis Lua Script가 어떻게 락 없이 원자성을 보장하는지, 그리고 대역폭을 절약하기 위한 실행 메커니즘을 심층 분석합니다.

---

## 1. 원자성(Atomicity)의 원리: "Single Thread Event Loop"

Redis가 Lua Script를 처리하는 방식은 RDBMS의 트랜잭션과 근본적으로 다릅니다.

### 1.1 "Stop the World" 아키텍처
*   **Event Loop:** Redis는 들어오는 모든 요청을 하나의 큐(Queue)에 넣고, **단 하나의 스레드**가 순서대로 꺼내서 처리합니다.
*   **Atomic Execution:** Lua 스크립트가 실행되는 순간, Redis는 이 스크립트 처리에만 전념합니다. 스크립트가 끝날 때까지 **다른 클라이언트의 어떠한 명령어(심지어 `PING` 조차)**도 개입할 수 없습니다.
*   **의미:** `GET`으로 값을 읽고 `SET`으로 값을 바꾸는 사이에, 다른 누군가가 값을 변경할 가능성(Race Condition)이 **물리적으로 0%**입니다. 이것이 바로 락(Lock) 없이 동시성을 제어하는 비결입니다.

### 1.2 왜 빠른가? (Zero Overhead)
*   **No Lock:** 락을 획득하고 해제하는 과정이 없습니다.
*   **No Context Switching:** CPU가 여러 스레드를 왔다 갔다 하며 낭비하는 시간이 없습니다.
*   **No Network I/O:** 스크립트 내부에서의 데이터 접근은 메모리 직접 접근이므로 지연 시간이 거의 없습니다.

### 1.3 주의사항: 블로킹(Blocking)의 위험
*   **양날의 검:** 원자성은 강력하지만, 스크립트가 1초라도 걸리면 **전 세계의 모든 서비스가 1초 동안 멈춥니다.**
*   **제약:** Lua 스크립트 내부에서는 절대 **오래 걸리는 연산(O(N) 탐색, 외부 API 호출, 무한 루프)**을 해서는 안 됩니다.

## 2. 실행 최적화: SCRIPT LOAD & EVALSHA

스크립트 본문을 매번 네트워크로 보내는 것은 엄청난 낭비입니다.

### 2.1 메커니즘
1.  **SCRIPT LOAD:** 애플리케이션 시작 시 스크립트 본문(`local key...`)을 Redis에 한 번 전송합니다.
2.  **SHA-1 반환:** Redis는 스크립트를 컴파일하여 캐싱하고, 그 내용의 해시값(`abc1234...`)을 반환합니다.
3.  **EVALSHA:** 이후 클라이언트는 긴 본문 대신 **40Byte짜리 짧은 해시값**만 보내서 실행합니다.
    *   `EVALSHA abc1234... 1 key1 arg1`

### 2.2 네트워크 대역폭 절약 효과
*   **상황:** 1KB짜리 복잡한 비즈니스 로직 스크립트를 초당 1만 번 호출.
*   **EVAL:** 1KB * 10,000 = **10MB/sec** 트래픽 발생. (네트워크 병목)
*   **EVALSHA:** 40Byte * 10,000 = **400KB/sec** 트래픽 발생. (**약 25배 절약**)

### 2.3 Behind the Scenes: Lua JIT & Replication
`EVALSHA`의 이점은 네트워크뿐만이 아닙니다.
*   **CPU 효율 (JIT):** `SCRIPT LOAD` 시 Redis는 Lua 코드를 **바이트코드(Bytecode)**로 컴파일하여 메모리에 적재합니다. `EVALSHA`는 파싱 과정 없이 이미 컴파일된 함수를 즉시 호출하므로 CPU 사이클을 절약합니다.
*   **Replication 대역폭:** 마스터에서 실행된 무거운 스크립트를 슬레이브로 복제할 때도, 스크립트 본문 전체를 보내는 대신 `EVALSHA` 명령어만 전파하여 내부망 대역폭을 아낄 수 있습니다.

## 3. 결정적 실행: The Pure Function Rule

Redis는 스크립트가 **"언제, 어디서 실행되든 항상 같은 결과"**를 내야 한다고 가정합니다. 이는 **복제(Replication) 메커니즘**의 특성 때문입니다.

### 3.1 왜 순수함수여야 하는가? (Script Replication)
Redis의 기본 복제 방식은 **"스크립트 소스 코드를 슬레이브로 보내서 재실행(Re-execution)"**하는 것입니다.

*   **이유 (대역폭 절약):** 스크립트 코드는 1KB인데, 그 스크립트가 생성한 데이터 변경(`SET` 1만 번)이 10MB라면? 10MB를 보내는 것보다 1KB 코드를 보내서 각자 실행하게 하는 것이 네트워크 효율적이기 때문입니다.
*   **제약:** 슬레이브가 코드를 받아서 **각자 실행**했을 때 마스터와 **똑같은 결과**가 나와야만 데이터 정합성이 유지됩니다. 그래서 스크립트는 무작위성이 없는 **순수 함수**여야 합니다.

### 3.2 대안: Effect Replication (결과 복제)
만약 `TIME`이나 `RANDOM`을 꼭 써야 한다면?
*   **Effect Replication:** 스크립트를 재실행하지 않고, 스크립트가 만들어낸 **최종 결과 명령어(`SET key val`)들만** 골라서 슬레이브로 보냅니다. (Redis 3.2+)
*   **장점:** 순수 함수 제약에서 벗어날 수 있음.
*   **단점:** 결과 데이터가 많으면 네트워크 대역폭 사용량이 급증함.

### 3.3 Engineering Principle: Pure Function (순수 함수)
여기서 말하는 '순수 함수'는 "데이터를 변경하지 않는다(No Side Effect)"는 뜻이 아닙니다. Redis 스크립트는 당연히 데이터를 변경합니다. Redis가 요구하는 순수성은 **"결정론적 부수 효과(Deterministic Side Effect)"**입니다.

*   **정의:** 입력(`KEYS`, `ARGV`)과 초기 데이터 상태가 같다면, 스크립트가 수행하는 데이터 변경(`SET`, `INCR`) 결과도 **언제나, 어디서나(Master/Slave) 100% 동일해야 한다**는 원칙입니다.
*   **위반 시 시나리오:**
    1.  스크립트 내부에 `local t = redis.call('TIME')` 사용.
    2.  **Master:** 12:00:00에 실행 -> `SET key "12:00:00"` 저장.
    3.  **Slave:** 스크립트가 복제되어 0.1초 뒤 재실행 -> `SET key "12:00:00.1"` 저장.
    4.  **결과:** Master와 Slave의 데이터가 달라지는 **심각한 데이터 불일치(Inconsistency)** 발생.
