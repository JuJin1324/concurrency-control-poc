# [Deep Dive] Redis Lua Script Architecture Patterns

**Parent Document:** [Lua Script 운영 가이드](../lua-script-ops.md)

Redis Lua Script는 락(Lock) 없이 원자적 연산을 수행하여 극한의 성능(TPS)을 제공하는 엔진입니다. 하지만 Redis의 고속 처리를 영속 저장소인 RDB가 따라가지 못하는 **"속도 불일치"** 문제를 해결하는 것이 아키텍처의 핵심입니다.

---

## 1. 엔터프라이즈 도입 사례 분석 (Case Studies)

### 1.1 티켓링크/인터파크: "느려도 확실하게" (동기식 저장)
**전략:** 원자적 좌석 선점 & 동기식 방어 (Synchronous)

*   **상황 (Scenario):** 아이유 콘서트 티켓팅. 수십만 명이 동시에 'A열 1번' 좌석을 클릭합니다.
*   **문제:**
    *   RDBMS의 락(`SELECT FOR UPDATE`)을 쓰면 DB가 터집니다.
    *   Redis만 믿고 "성공"을 줬다가, 나중에 DB 저장할 때 에러가 나면? -> **"예매 성공했다고 해서 갔는데 표가 없대요"**라는 대형 사고가 터집니다.
*   **해결책 (프로세스):**
    1.  **Redis 선점 (Lua):** Lua 스크립트로 `GET`(좌석확인) + `SISMEMBER`(중복확인) + `SADD`(점유)를 0.001초 만에 처리합니다.
    2.  **DB 저장 (Blocking):** Redis 성공 직후, 유저에게 응답을 주기 **전에** DB `INSERT`를 시도합니다.
    3.  **응답:** DB 저장까지 완벽하게 끝나야 비로소 유저 화면의 로딩이 멈추고 "예매 성공"이 뜹니다.
*   **Trade-off:**
    *   DB가 느려지면 유저 대기 시간도 길어집니다.
    *   하지만 **"데이터 정합성(Consistency)"**을 속도보다 우선시한 결정입니다. 결제와 관련된 문제는 타협할 수 없기 때문입니다.

### 1.2 우아한형제들 (배달의민족): "일단 주고 나중에 정리" (비동기식 저장)
**전략:** Eventual Consistency (비동기) & Write-Behind

*   **상황 (Scenario):** 선착순 1,000명 치킨 쿠폰 이벤트.
*   **문제:**
    *   이벤트는 **"속도감"**이 생명입니다. 클릭했는데 1초 동안 뺑글이가 돌면 유저는 "렉 걸렸다"고 생각하고 이탈합니다.
    *   티켓팅처럼 DB 저장을 기다려줄 여유가 없습니다.
*   **해결책 (프로세스):**
    1.  **Redis 선점 (Lua):** `SCARD`(수량확인) + `SADD`(발급)로 0.001초 만에 당첨 여부를 결정합니다.
    2.  **즉시 응답:** Redis가 OK 하면, DB 저장 여부와 상관없이 유저에게 **바로 "축하합니다!" 팝업**을 띄웁니다.
    3.  **후처리 (Async):** 서버는 뒷단에서 Kafka나 큐(Queue)에 "유저 A에게 쿠폰 줬음"이라는 쪽지(Event)를 남깁니다.
    4.  **DB 저장:** 별도의 워커(Worker) 프로세스가 큐에서 쪽지를 하나씩 꺼내 DB에 기록합니다. (1초 뒤에 저장되든 10초 뒤에 저장되든 유저는 모릅니다.)
*   **Trade-off:**
    *   아주 드물게 큐가 유실되거나 DB 저장이 실패하면, 유저는 쿠폰함에서 쿠폰을 못 찾을 수도 있습니다.
    *   이를 막기 위해 **보상 트랜잭션**이나 **재시도 로직** 등 운영 비용이 더 들어갑니다. 하지만 **"최상의 사용자 경험(UX)"**을 위해 이를 감수합니다.

### 1.3 Deep Analysis: 성능의 경제학 (The Economics of Performance)
결국 분산 시스템 아키텍처는 **"Redis로 벌어들인 성능(Speed Gain)이라는 자원을 어디에 재투자할 것인가?"**에 대한 의사결정입니다.

*   **티켓링크의 선택 (Investment in Trust):**
    *   Redis로 얻은 압도적인 속도 여유분을 **"유저를 기다리게 하는 시간"**으로 지불하고, 대신 **"완벽한 정합성(Trust)"**을 샀습니다.
    *   "빨리 처리하는 것"보다 "실수하지 않는 것"이 비즈니스의 본질이기 때문입니다.
*   **배달의민족의 선택 (Investment in Experience):**
    *   Redis로 얻은 속도 효능감을 **"고객 경험(UX)"**에 100% 몰빵했습니다.
    *   그리고 그로 인해 구멍 난 정합성은, 백엔드 엔지니어링(보상 트랜잭션, 배치)이라는 **"추가 비용"**을 지불하여 메꾸는 방식을 택했습니다.

## 2. DB 동기화 전략 비교 분석

Redis의 미친 속도를 느린 DB에 어떻게 맞출 것인가?

| 전략 | 메커니즘 | 장점 | 단점 | 권장 상황 |
| :--- | :--- | :--- | :--- | :--- |
| **Synchronous** (동기) | Lua 성공 -> **DB 저장 대기** -> 응답 | 데이터 유실 0%. 구현이 단순함. | DB 속도만큼 전체 응답이 느려짐. DB 장애 시 서비스 중단. | 결제, 좌석 선점, 금융 거래 |
| **Asynchronous** (비동기) | Lua 성공 -> **이벤트 발행** -> 즉시 응답 | 최고의 응답 속도. DB가 죽어도 서비스 가능. | 데이터 불일치 가능성 있음. 복구 로직이 복잡함. | 선착순 쿠폰, 티켓팅(비동기 전환 시) |
| **Micro-Batch** (집계) | 메모리에서 카운팅만 하다가 **1초~1분 주기**로 저장 | DB 부하 극단적 감소 (1만 번 클릭 -> 1번 DB 업데이트). | 장애 시 최근(1초~1분) 데이터 유실. 이력(History) 추적 불가. | 조회수, 좋아요, 랭킹 점수 |

### 2.1 Advanced: Micro-Batch vs Redis Stream
대량의 데이터를 처리하는 두 가지 접근 방식의 차이점입니다.

| 구분 | **Micro-Batch (Snapshot)** | **Redis Stream (Log)** |
| :--- | :--- | :--- |
| **철학** | "과정은 버리고 **결과(숫자)**만 남긴다." | "모든 **이력(History)**을 순서대로 남긴다." |
| **동작** | `INCR`로 숫자만 계속 바꿈. | `XADD`로 이벤트를 계속 쌓음. |
| **장점** | 압도적인 성능. 메모리 절약. | 완벽한 이력 추적. Consumer Group 병렬 처리. |
| **단점** | 누가 언제 눌렀는지 모름. | 데이터가 쌓일수록 메모리 증가. DB 저장 부하 여전함. |
| **Use Case** | **유튜브 조회수, 인스타 좋아요** | **채팅 내역, 주문 로그, 활동 추적** |

## 3. 정합성 보장 아키텍처 (Reconciliation)



비동기 방식(Asynchronous)을 선택했다면, Redis와 DB 간의 데이터 불일치는 단순히 '어쩌다 발생하는 사고'가 아니라 **'언젠가는 반드시 발생할 수 있는 리스크'**로 간주하고 **설계 단계에서부터 대비**해야 합니다. 이를 사후에 바로잡는 과정이 바로 **Reconciliation(정합성 보정)**입니다.





### 3.1 Full-Scan Comparison (전수 비교)

가장 확실한 방법으로, 보통 트래픽이 적은 새벽 시간대에 배치(Batch)로 수행합니다.



1.  **Redis 명단 추출:** 당첨자 목록이 저장된 Redis Set에서 유저 ID 목록을 가져옵니다.

    *   **주의:** 데이터가 수만 건 이상이라면 `SMEMBERS`는 Redis를 블로킹하므로 **`SSCAN`** 명령어로 쪼개서 가져와야 합니다.

2.  **DB 명단 추출:** DB에서 동일한 이벤트의 주문/발급 테이블을 조회하여 명단을 가져옵니다.

3.  **차집합 계산 (The Diff):** `Redis 명단 - DB 명단`을 수행합니다. 여기에 속한 유저들은 **"Redis에서는 성공했으나 DB 저장 과정에서 유실된"** 케이스입니다.

4.  **자동 복구:** 누락된 유저들에 대해 DB `INSERT`를 재시도하고, 필요시 유저에게 알림을 발송합니다.



### 3.2 Saga Pattern (보상 트랜잭션)

실시간으로 실패를 감지했을 때 즉시 원상복구 하는 전략입니다.



*   **동작:** 비동기 워커가 DB 저장을 시도했으나 **영구적인 실패**(예: 유저 탈퇴, 제약 조건 위반)가 발생한 경우.

*   **복구:** 다시 Redis로 돌아가서 해당 유저를 `SREM`으로 삭제하고 재고를 `INCR` 시켜야 합니다. 이를 방치하면 Redis 재고와 실제 판매 수량이 영원히 맞지 않게 됩니다.



---







### 💡 운영 팁: "SSCAN은 선택이 아닌 필수"



Reconciliation 배치를 짤 때 가장 흔히 하는 실수는 `SMEMBERS`나 `KEYS *`를 사용하는 것입니다.







*   **SMEMBERS (위험/Blocking):** "모든 멤버를 한 번에 다 가져와!"라고 명령하는 것과 같습니다. 데이터가 100만 건이라면 Redis는 이를 퍼 올리는 수 초 동안 **다른 모든 요청을 처리하지 못하고 멈춥니다.** 이는 즉시 서비스 장애로 이어집니다.



*   **SSCAN (안전/Non-blocking):** "한 번에 100개씩만 가져오고, 다음 위치(Cursor)를 알려줘"라고 요청하는 방식입니다. 한 번의 요청이 극히 짧으므로 실시간 서비스에 지장을 주지 않습니다.







**[비유]** 편의점 알바에게 "재고 조사하게 창고 물건 다 꺼내와!"라고 하면 손님은 계산을 못 하고 줄을 서야 하지만(**SMEMBERS**), "손님 계산 틈틈이 10개씩만 세어봐"라고 하면 서비스는 계속 유지됩니다(**SSCAN**).
