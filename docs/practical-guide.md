# 실무 적용 가이드: 동시성 제어 전략 선택하기

**작성일:** 2026-01-27
**기반 데이터:** Performance Test Report v2 (High/Extreme Load)

---

## "어떤 상황에 어떤 방법을 써야 할까?"

### 1. Pessimistic Lock (비관적 락)
**추천 상황:**
- **데이터 정합성이 최우선**일 때 (금전 거래, 핵심 재고).
- **충돌이 빈번**하게 발생할 것으로 예상될 때.
- **안정성이 성능보다 중요할 때.** (가장 예측 가능한 성능을 보임)

**장점:** 구현이 간단하고(`@Lock`), 데이터 정합성이 확실함.
**단점:** DB 커넥션을 오래 점유할 수 있음.

---

### 2. Optimistic Lock (낙관적 락)
**추천 상황:**
- **읽기 요청이 쓰기보다 압도적으로 많을 때.**
- **웹/API 환경** (네트워크 지연이 자연스러운 경합 분산 효과를 줌).
- DB 락을 걸지 않고 애플리케이션 레벨에서 처리하고 싶을 때.

**장점:** 별도의 락 인프라(Redis 등)가 필요 없고 DB 락 부하가 없음.
**단점:** 충돌 발생 시 **재시도(Retry) 로직**을 직접 구현해야 함.

---

### 3. Redis Distributed Lock (분산 락)
**추천 상황:**
- **분산 DB / MSA 환경**에서 여러 서비스가 동시에 자원에 접근할 때.
- **복잡한 비즈니스 로직**을 락으로 보호해야 할 때 (단순 재고 차감 X).

**주의사항:**
- **대규모 트래픽(Spike) 시 병목 발생 가능성 높음.** (락 획득 대기 오버헤드)
- 단순 재고 차감용으로는 오버헤드가 큼.

---

### 4. Redis Lua Script
**추천 상황:**
- **선착순 이벤트, 티켓팅** 등 극한의 트래픽을 처리해야 할 때.
- **최고의 성능(Throughput)과 효율성**이 필요할 때.

**장점:** Network RTT를 최소화(1회 호출)하고 락 없이 원자성 보장.
**단점:** **DB와의 데이터 동기화(Consistency) 문제**를 해결해야 함 (Write-Behind 등).

---

## 실무 도입 체크리스트

1.  **트래픽 패턴 분석:**
    - 읽기가 많은가? 쓰기가 많은가? -> 읽기 많으면 Optimistic
    - 특정 시간에 몰리는가(Spike)? -> Lua Script 고려
2.  **인프라 환경:**
    - Redis를 이미 사용 중인가?
    - DB가 병목 지점인가? -> DB Lock 회피 (Lua Script)
3.  **성능 요구사항:**
    - 응답 속도가 중요한가? -> Optimistic / Lua
    - 안정성이 중요한가? -> Pessimistic

## 결론
> **"Start Simple, Optimize Later."**

1. **Pessimistic Lock**으로 시작하여 비즈니스 안정성을 확보하세요.
2. 성능 이슈가 발생하면 **Lua Script** (선착순) 또는 **Optimistic Lock** (일반 웹)으로 고도화하세요.
3. **Redis Distributed Lock**은 분산 환경 제어가 필수적인 경우에만 신중하게 도입하세요.
