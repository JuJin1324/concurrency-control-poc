# 실무 적용 가이드: 동시성 제어 전략 선택하기

**작성일:** 2026-01-27

---

## "어떤 상황에 어떤 방법을 써야 할까?"

본 PoC의 테스트 결과를 바탕으로, 상황별 최적의 동시성 제어 전략을 제안합니다.

### 1. Pessimistic Lock (비관적 락)
**추천 상황:**
- **데이터 정합성이 최우선**일 때 (금전 거래, 핵심 재고).
- **충돌이 빈번**하게 발생할 것으로 예상될 때.
- **단일 DB** 또는 마스터 DB 위주로 트래픽을 처리할 때.

**장점:** 구현이 간단하고(`@Lock`), 데이터 정합성이 확실함.
**단점:** DB 커넥션을 오래 점유할 수 있어 트래픽 폭주 시 DB 병목이 될 수 있음.

---

### 2. Optimistic Lock (낙관적 락)
**추천 상황:**
- **충돌이 가끔** 발생할 때 (대부분의 일반적인 웹 서비스).
- **읽기 요청이 쓰기보다 압도적으로 많을 때.**
- DB 락을 걸지 않고 애플리케이션 레벨에서 처리하고 싶을 때.

**장점:** 별도의 락 인프라(Redis 등)가 필요 없고 DB 락 부하가 없음.
**단점:** 충돌 발생 시 **재시도(Retry) 로직**을 직접 구현해야 하며, 충돌이 잦으면 재시도로 인해 성능이 급격히 저하됨.

---

### 3. Redis Distributed Lock (분산 락)
**추천 상황:**
- **분산 DB / MSA 환경**에서 여러 서비스가 동시에 자원에 접근할 때.
- **DB 부하를 줄이고 싶을 때** (락 관리를 Redis에 위임).
- 락 획득 대기 시간(Timeout)을 세밀하게 제어하고 싶을 때.

**장점:** DB 스케일아웃 환경에서도 동시성 제어 가능.
**단점:** Redis 인프라 관리 필요, 구현 복잡도 증가(Redisson 등 라이브러리 의존), 네트워크 지연(RTT) 발생.

---

### 4. Redis Lua Script
**추천 상황:**
- **극한의 성능(TPS)**이 필요할 때 (선착순 이벤트, 티켓팅).
- **복잡한 연산을 원자적으로** 처리해야 할 때.
- 재고 차감 등 단순 수치 연산이 주를 이룰 때.

**장점:** Network RTT를 최소화(1회 호출)하고 Redis의 싱글 스레드 특성을 이용해 락 없이 원자성 보장.
**단점:** **DB와의 데이터 동기화(Consistency) 문제**를 해결해야 함 (Write-Behind, Batch Update 등 추가 구현 필요). 디버깅이 어려움.

---

## 실무 도입 체크리스트

1.  **트래픽 패턴 분석:**
    - 읽기가 많은가? 쓰기가 많은가?
    - 특정 시간에 몰리는가(Spike)? 꾸준한가(Constant)?
2.  **인프라 환경:**
    - Redis를 이미 사용 중인가? (운영 비용 고려)
    - DB가 병목 지점인가?
3.  **비즈니스 중요도:**
    - 재고가 -1이 되어도 되는가? (절대 안 된다면 Pessimistic/Redis Lock)
    - 사용자에게 "잠시 후 다시 시도해주세요"라고 해도 되는가? (Optimistic Lock)

## 결론
> **"Premature optimization is the root of all evil."**

처음부터 복잡한 **Redis Lock**이나 **Lua Script**를 도입하기보다, **Pessimistic Lock**으로 시작하여 안정성을 확보한 후, 성능 테스트(k6 등)를 통해 병목이 확인되면 그때 **Redis** 기반 솔루션으로 고도화하는 것을 권장합니다.
