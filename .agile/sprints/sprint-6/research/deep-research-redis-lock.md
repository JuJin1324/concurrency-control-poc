# **Redis Distributed Lock: 실무 도입 사례 분석 및 운영 고도화 전략**

## **1. 서론: 분산 시스템에서의 동시성 제어 패러다임**

현대 IT 인프라가 모놀리식(Monolithic) 아키텍처에서 마이크로서비스 아키텍처(MSA)로 전환됨에 따라, 시스템 전반의 데이터 일관성과 무결성을 보장하는 방법론은 중대한 도전에 직면했다. 과거 단일 데이터베이스의 ACID 트랜잭션이나 단일 프로세스 내의 synchronized 블록, ReentrantLock 등으로 해결 가능했던 동시성 제어 문제는, 물리적으로 분리된 서버와 네트워크 파티션, 이기종 데이터 저장소가 혼재하는 분산 환경에서 더 이상 유효하지 않다. 특히 수평적으로 확장된(Scale-out) 다수의 서버 인스턴스가 단일 공유 자원(재고, 계좌 잔액, 좌석 등)에 동시에 접근할 때 발생하는 경쟁 상태(Race Condition)는 비즈니스 로직의 오작동뿐만 아니라 금전적 손실, 사용자 신뢰 하락으로 직결되는 치명적인 위험 요소이다.1

이러한 배경 속에서 Redis Distributed Lock(분산 락)은 분산 환경에서의 상호 배제(Mutual Exclusion)를 구현하기 위한 사실상의 표준 기술로 자리 잡았다. Redis는 디스크 기반의 데이터베이스 락(Pessimistic Lock) 대비 월등히 높은 처리량(Throughput)과 낮은 지연 시간(Latency)을 제공하며, 싱글 스레드 아키텍처가 보장하는 명령어 단위의 원자성(Atomicity) 덕분에 복잡한 동시성 제어 로직을 단순화할 수 있다. 그러나 Redis 분산 락은 '은탄환(Silver Bullet)'이 아니다. 네트워크 불안정성, 클럭 편향(Clock Skew), 가비지 컬렉션(GC)으로 인한 프로세스 중단 등 분산 시스템 고유의 불확실성은 락의 안전성(Safety)과 활동성(Liveness)을 위협한다.2

본 리포트는 우아한형제들(배달의민족), 토스, 쿠팡, 그랩(Grab), 우버(Uber) 등 대규모 트래픽을 처리하는 국내외 선도 기업들의 실제 Redis 분산 락 도입 사례를 심층 분석한다. 각 기업이 직면한 비즈니스 문제와 이를 해결하기 위해 Redis 락을 선택한 기술적 의사결정 과정, 그리고 운영 중에 겪은 장애 경험과 이를 통해 축적된 노하우를 상세히 기술한다. 나아가 Redis 토폴로지 구성, 타임아웃 설정, 모니터링 체계 구축 등 실무 운영자가 반드시 숙지해야 할 고도화된 운영 전략을 제시함으로써, 안정적이고 효율적인 분산 락 시스템 구축을 위한 가이드를 제공하는 것을 목표로 한다.

## **2. Redis 분산 락의 기술적 메커니즘과 구현 패턴 심층 분석**

실무 사례를 분석하기에 앞서, Redis 분산 락의 핵심이 되는 기술적 메커니즘과 주요 클라이언트 구현체의 동작 원리를 이해할 필요가 있다. 이는 단순한 도구의 사용을 넘어, 장애 상황에서의 동작을 예측하고 최적의 설정을 도출하는 기반이 된다.

### **2.1 스핀 락(Spin Lock) 모델과 Lettuce의 한계**

초기 Redis 분산 락 구현은 SETNX (SET if Not eXists) 명령어에 의존하는 스핀 락 형태가 주를 이루었다. 이는 클라이언트가 락 획득을 시도하고, 실패할 경우 일정 시간 대기 후 다시 시도하는 단순한 구조다.

- **메커니즘**: 클라이언트는 SET lock_key unique_id NX PX 30000과 같은 명령을 전송한다. 여기서 NX는 키가 존재하지 않을 때만 저장함을, PX는 락의 자동 만료 시간(TTL)을 밀리초 단위로 설정함을 의미한다. 성공 시 락을 획득하고, 실패 시 루프를 돌며 재시도한다.3
- **문제점**: 스핀 락 방식은 락 경쟁이 치열한 상황에서 치명적인 성능 저하를 유발한다. 락을 획득하지 못한 수백, 수천 개의 스레드가 지속적으로 Redis에 SET 명령을 전송함으로써 Redis 서버의 CPU 사용률을 급격히 상승시킨다. 이는 정작 락을 획득하여 비즈니스 로직을 처리해야 할 요청의 응답 속도마저 저하시키는 '형님 리스크(Thundering Herd)' 문제를 야기할 수 있다.4

### **2.2 Pub/Sub 기반 이벤트 모델과 Redisson의 혁신**

Redisson 라이브러리는 스핀 락의 비효율성을 극복하기 위해 Redis의 Pub/Sub 기능을 활용한 이벤트 기반 락을 구현했다.

- **동작 원리**: 락 획득에 실패한 클라이언트는 무의미한 재시도(Spin)를 멈추고, 해당 락의 해제 이벤트를 수신하는 채널을 구독(Subscribe)한 뒤 대기 상태(Sleep)로 전환된다. 락을 보유한 클라이언트가 작업을 마치고 락을 해제(UNLOCK)하면, Pub/Sub 채널을 통해 대기 중인 클라이언트들에게 신호를 보내고, 신호를 받은 클라이언트들만이 다시 락 획득을 시도한다.
- **Lua 스크립트를 통한 원자성**: Redisson은 락의 획득, 검증, 해제, TTL 갱신 등 복합적인 연산을 Lua 스크립트로 캡슐화하여 서버 측에서 원자적으로 실행한다. 이는 네트워크 왕복 시간(RTT)을 줄이고, 명령어 사이의 경합 상태를 방지하는 핵심 기술이다.1

### **2.3 Redlock 알고리즘: 이론적 이상과 현실적 제약**

단일 Redis 노드의 장애(SPOF) 위험을 회피하기 위해 제안된 Redlock 알고리즘은 N개의 독립적인 Redis 마스터 노드에 락을 분산 저장한다.

- **합의 알고리즘**: 클라이언트는 N개의 노드(보통 5개)에 순차적으로 락 획득을 시도하며, 과반수(N/2 + 1) 이상의 노드에서 락을 획득하고, 그 소요 시간이 락 유효 시간보다 짧을 때 비로소 락 획득 성공으로 간주한다.
- **비판과 한계**: 분산 시스템 전문가 Martin Kleppmann은 Redlock이 클럭 동기화(Clock Synchronization)에 의존적이며, 펜싱 토큰(Fencing Token)과 같은 단조 증가 ID를 제공하지 못해 GC Pause와 같은 상황에서 데이터 정합성을 보장하지 못한다고 비판했다. 이러한 이론적 논쟁과 구현의 복잡성으로 인해, 실제 현업에서는 Redlock보다는 단일 클러스터와 애플리케이션 레벨의 낙관적 락을 병행하는 실용적 접근이 선호되는 추세다.2

## **3. 엔터프라이즈 도입 사례 심층 분석**

국내외 주요 테크 기업들은 각자의 비즈니스 도메인 특성에 맞춰 Redis 분산 락을 다양하게 변주하여 적용하고 있다. 이들 사례는 단순한 기술 도입을 넘어, 트래픽 패턴과 데이터 중요도에 따른 아키텍처 설계의 모범 답안을 제시한다.

### **3.1 사례 1: 우아한형제들 (배달의민족) - 선착순 이벤트 시스템**

배달의민족의 '선착순 쿠폰 이벤트'는 짧은 시간 내에 폭발적인 트래픽이 집중되는 대표적인 고성능 동시성 제어 사례다.

- **도메인 및 문제 정의**:
- **서비스명**: 선착순 할인 쿠폰 발급 시스템
- **핵심 과제**: 정해진 시간에 수만 명의 사용자가 동시에 접속하여 한정된 수량(예: 1,000장)의 쿠폰 발급을 요청한다. 초과 발급(Over-issuance)은 금전적 손실과 사용자 불만을 야기하므로 절대적으로 방지해야 한다. 기존 RDB의 행 단위 잠금(Row Lock) 방식은 DB 커넥션 풀 고갈과 데드락 위험으로 인해 초당 수만 건의 요청을 처리하기에 역부족이었다.1
- **구현 방식**: **Redis Set 자료구조와 Redisson의 조화**
- **아키텍처**: Redis를 주 저장소(Primary Storage)로 활용하여 실시간 재고를 관리하고, RDB는 비동기적으로 이력을 저장하는 구조를 채택했다.
- **자료구조 활용**: 단순한 분산 락(LOCK -> GET -> DECR -> UNLOCK) 패턴 대신, Redis의 Set 자료구조를 활용했다. 사용자 ID를 Set에 넣는 SADD 명령어의 반환값을 통해 중복 발급 여부를 원자적으로 체크하고, SCARD 명령어로 현재 발급량을 실시간 확인했다. 이는 락을 획득하고 해제하는 오버헤드조차 제거한 고성능 패턴이다.1
- **Redisson 적용**: 재고 관리 외에 동시성 제어가 필요한 로직(예: 주문 데이터 생성)에서는 Redisson 클라이언트를 도입했다. 특히, **Watchdog** 기능을 활성화하여 비즈니스 로직 수행 중 예상치 못한 지연이 발생하더라도 락이 조기에 만료되어 정합성이 깨지는 것을 방지했다.
- **설정값**: waitTime은 트래픽 스파이크 시 대기열 폭증을 막기 위해 짧게(예: 2~3초) 설정하고, leaseTime은 기본 Watchdog 설정(30초)을 따르되 명시적 TTL이 필요한 경우 넉넉하게(예: 10분) 설정하여 락 누수 시 자동 복구를 도모했다.
- **성능 지표 및 효과**:
- RDB 락 사용 대비 처리량(TPS)이 수십 배 향상되었으며, DB 부하를 획기적으로 줄여 전체 주문 시스템의 안정성을 확보했다.
- Redis 클러스터를 이벤트 전용으로 분리함으로써, 이벤트 트래픽이 메인 커머스 로직에 영향을 주지 않도록 격리(Isolation)했다.
- **운영 노하우 및 장애 경험**:
- **장애 경험**: 초기 Lettuce 사용 시 스핀 락으로 인한 Redis CPU 100% 점유 현상을 겪었다. 이를 Redisson으로 교체하며 Pub/Sub 모델로 전환, CPU 사용률을 안정화했다.1
- **Fallback 전략**: Redis 장애 발생 시, 이벤트를 일시 중단하고 '잠시 후 다시 시도해주세요' 메시지를 노출하는 Fail-Safe 전략을 취했다. 이는 이벤트의 특성상 데이터 정합성이 가용성보다 중요했기 때문이다.

### **3.2 사례 2: 토스 (Toss) - 결제 및 포인트 정합성 제어**

토스는 금융 플랫폼으로서 데이터의 정확성이 생명이다. 특히 사용자의 실수나 네트워크 지연으로 인한 중복 요청(따닥)을 방어하는 것이 핵심이다.

- **도메인 및 문제 정의**:
- **서비스명**: 포인트 지급 및 결제 시스템
- **핵심 과제**: 사용자가 '포인트 받기' 버튼을 빠르게 두 번 누르거나, 결제 요청이 네트워크 타임아웃 후 재전송될 때 중복으로 포인트가 지급되거나 결제가 승인되는 것을 막아야 한다.
- **구현 방식**: **경량화된 Lettuce 래퍼와 하이브리드 락킹**
- **구현 특징**: 토스는 Redisson의 무거움 대신, Lettuce를 기반으로 SETNX를 활용한 경량 분산 락을 직접 구현하여 사용하기도 했다. Kotlin의 고차 함수 기능을 활용해 acquireAndRunLock(key) { logic } 형태의 래퍼를 만들어, 락의 획득과 해제(finally 블록)를 강제함으로써 개발자의 실수를 원천 차단했다.3
- **키 설계**: 락의 키를 lock:userId:eventId와 같이 사용자 ID와 이벤트 ID를 조합하여 락의 범위를 최소화(Granularity 최적화)했다. 이는 전체 시스템의 동시 처리량을 극대화하는 핵심 전략이다.
- **낙관적 락과의 병행**: Redis 락을 1차 방어선으로 사용하되, 최종 데이터베이스 반영 시에는 JPA의 @Version을 이용한 낙관적 락(Optimistic Lock)을 2차 방어선으로 구축했다. Redis 락이 만료되거나 장애로 인해 뚫리더라도, DB 차원에서 버전 충돌을 감지하여 롤백시킴으로써 데이터 무결성을 최후까지 보장했다.7
- **운영 노하우**:
- **락 해제 검증**: 락을 해제할 때 반드시 자신의 UUID 토큰과 비교하는 Lua 스크립트를 사용하여, 타임아웃으로 인해 다른 프로세스가 획득한 락을 실수로 해제하는 사고를 방지했다.
- **Fallback**: Redis 장애 시에는 DB 락으로 전환하기보다는, 사용자에게 재시도를 요청하거나 대기열로 안내하는 방식을 택하여 시스템 복잡도를 낮췄다.

### **3.3 사례 3: 쿠팡 (Coupang) - 대규모 장애와 SPOF의 교훈**

쿠팡의 사례는 분산 락 시스템이 단일 장애 지점(SPOF)이 되었을 때의 위험성을 극명하게 보여준다.

- **사건 개요 (2019년)**:
- 모든 상품이 '품절'로 표시되어 주문이 불가능한 대규모 장애가 발생했다.
- **원인**: 재고 상태를 관리하고 분산 락을 제공하던 Redis 시스템에 버그 및 메모리 과부하가 발생했고, 이로 인해 재고 확인 로직이 락을 획득하지 못하거나 타임아웃으로 실패하면서 방어 로직에 의해 모든 상품을 '재고 없음(0)'으로 처리했다.8
- **운영적 교훈 및 개선**:
- **SPOF 제거의 중요성**: Redis 분산 락에 대한 의존도가 너무 높으면 Redis 장애가 곧 서비스 전체의 장애로 이어진다. 쿠팡은 이후 아키텍처에서 캐시 레이어(Redis)와 영속 레이어(RDB) 간의 결합을 느슨하게 하고, Redis 장애 시 RDB를 직접 조회하거나 서킷 브레이커(Circuit Breaker)를 통해 기본 동작을 수행하도록 복원력을 강화했다.
- **Reconciliation(대사) 프로세스**: 분산 락을 사용하더라도 미세한 타이밍 이슈나 장애로 인해 Redis 재고와 DB 재고가 불일치할 수 있다. 주기적인 배치 작업을 통해 두 데이터 소스의 정합성을 맞추는 대사 프로세스의 중요성이 강조되었다.9

### **3.4 사례 4: 그랩 (Grab) & 우버 (Uber) - 배차 시스템과 락의 한계 극복**

글로벌 모빌리티 기업들은 '배차'라는 복잡한 프로세스에서 분산 락의 한계를 경험하고 새로운 아키텍처로 진화했다.

- **도메인 및 문제 정의**:
- **서비스명**: 차량 호출 및 배차 시스템 (Rider Allocation)
- **핵심 과제**: 한 명의 드라이버에게 동시에 두 명의 승객이 배정되지 않아야 하며, 배차 프로세스는 수락-검증-할당이라는 긴 시간(Long-running)이 소요되는 워크플로우다.
- **기술적 진화**:
- **초기**: Redis 기반의 분산 락을 사용했으나, 배차 로직이 복잡해지면서 락 점유 시간이 길어졌고, 적절한 TTL 설정이 난해해졌다. TTL을 너무 길게 잡으면 드라이버 앱이 죽었을 때 락이 안 풀려 다른 배차를 못 받고, 짧게 잡으면 배차 처리 도중 락이 풀려 중복 배차가 발생했다.10
- **Uber의 DLM**: 우버는 이를 해결하기 위해 자체적인 분산 락 매니저(DLM)를 개발했다. Redis 대신 Ringpop(가십 프로토콜 기반 샤딩)과 Cassandra(상태 저장)를 결합하여, 네트워크 파티션 상황에서도 더 강력한 일관성을 제공하는 락 시스템을 구축했다.11
- **Grab의 Temporal 도입**: 그랩은 최근 Redis와 SQS를 조합한 복잡한 비동기 아키텍처를 버리고, **Temporal** 워크플로우 엔진을 도입했다. Temporal은 락을 거는 대신 워크플로우의 상태를 영속화하고 큐를 통해 순차 처리를 보장함으로써, **'락 없는 동시성 제어(Lock-free Concurrency)'**를 구현했다. 이는 락 누수나 TTL 관리의 고통에서 벗어나는 근본적인 해결책이 되었다.10

## **4. 운영 고도화 및 아키텍처 전략**

실무에서 Redis 분산 락을 안정적으로 운영하기 위해서는 단순한 API 호출을 넘어선 깊이 있는 아키텍처 설계와 운영 전략이 필요하다.

### **4.1 Redis 토폴로지별 락 안정성 비교 분석**

어떤 Redis 구성(Topology)을 사용하느냐에 따라 분산 락의 신뢰성 수준이 결정된다. 아래 표는 각 구성별 트레이드오프를 비교 분석한 것이다.

| **구성 방식** | **아키텍처 특징** | **분산 락 관점의 장점** | **분산 락 관점의 위험 요소 및 제약** |
| --- | --- | --- | --- |
| **Standalone** | 단일 마스터 노드 | • 구현이 가장 간단함
• 마스터 생존 시 강력한 일관성 보장 | • **SPOF**: 마스터 장애 시 락 서비스 전면 중단
• 확장성 제한 (CPU, Memory) |
| **Sentinel** | 마스터-슬레이브 + 자동 페일오버 | • 자동 복구(Failover)를 통한 고가용성(HA) 확보 | • **Split-Brain**: 마스터 장애 시, 락 데이터가 슬레이브로 복제되기 전 페일오버가 발생하면 새 마스터에서 락이 유실되어 중복 락 획득 가능성 존재 (비동기 복제 한계) |
| **Cluster** | 데이터 샤딩 + 자동 페일오버 | • 데이터 분산으로 대규모 트래픽 처리 가능
• 노드 추가를 통한 수평적 확장 용이 | • **Multi-key 연산 제약**: 락 키가 서로 다른 해시 슬롯에 배정될 경우 트랜잭션 처리가 복잡함
• Sentinel과 동일한 비동기 복제 지연 문제 내재 |

**[운영 권고안]**

- **일반적인 서비스**: **Sentinel** 또는 **Cluster** + **Redisson** 조합을 사용하되, 드물게 발생하는 락 유실(Failover 시점)을 감수하고 **DB 낙관적 락**이나 유니크 제약조건으로 최종 방어를 수행하는 것이 성능과 비용 효율성 측면에서 가장 합리적이다.
- **극도로 엄격한 정합성 요구 시**: Zookeeper나 Etcd와 같이 CP(Consistency & Partition Tolerance)를 보장하는 시스템을 사용하거나, Redis 상에서 **Redlock** 알고리즘을 구현해야 한다. 하지만 Redlock은 운영 복잡도가 높고 성능 저하가 커 실무 채택률은 낮은 편이다.

### **4.2 네트워크 지연(RTT)과 락 성능 상관관계**

분산 락의 성능은 Redis 처리 속도보다 클라이언트와 Redis 간의 네트워크 왕복 시간(RTT)에 의해 좌우된다.

- **영향 분석**: 락 획득(LOCK)과 해제(UNLOCK) 과정에서 최소 2번 이상의 네트워크 통신이 발생한다. 만약 클라이언트와 Redis가 서로 다른 가용 영역(AZ)이나 리전에 위치하여 RTT가 1ms에서 10ms로 증가한다면, 락 처리량은 이론적으로 1/10로 급감한다.
- **최적화 전략**:
- **Locality 확보**: Redis 클러스터와 애플리케이션 서버를 동일한 AZ에 배치하여 RTT를 1ms 미만으로 유지해야 한다.
- **Pipelining 활용**: 가능하다면 락 획득과 비즈니스 로직 수행에 필요한 데이터 조회를 파이프라이닝하여 네트워크 왕복 횟수를 줄인다.
- **Lua 스크립트**: 여러 명령어를 하나의 스크립트로 묶어 보내 통신 횟수를 1회로 단축한다.

### **4.3 락 누수(Lock Leak) 방지 및 TTL 설정 가이드**

락 누수는 시스템을 멈추게 하는 가장 흔한 장애 원인이다. 락을 획득한 프로세스가 해제하지 못하고 죽었을 때, TTL 설정이 미비하다면 해당 리소스는 영원히 잠기게 된다.

- **TTL 설정의 딜레마와 해결책**:
- TTL이 너무 짧으면: 로직 수행 중 락이 풀려 동시성 이슈 발생.
- TTL이 너무 길으면: 장애 발생 시 데드락 상태 지속.
- **해결책 (Redisson Watchdog)**: TTL을 명시적으로 설정하지 않으면 Redisson은 기본값 30초로 락을 설정하고, 백그라운드 스레드(Watchdog)가 10초마다(기본 설정의 1/3) 락의 TTL을 다시 30초로 연장한다. 프로세스가 정상 종료되면 Watchdog도 종료되어 갱신이 멈추고, 프로세스가 비정상 종료되면 30초 후 Redis에 의해 자동으로 락이 해제된다. 이것이 가장 안전한 설정이다.12
- **권장 설정**: 명시적 TTL을 사용해야 한다면, **(P99 로직 수행 시간) × 3 ~ 5** 정도로 설정하여 충분한 여유를 두어야 한다.

### **4.4 관측성(Observability) 확보: 모니터링 지표**

보이지 않는 락 경합은 시스템의 시한폭탄과 같다. Prometheus와 Grafana를 통해 다음 지표들을 반드시 모니터링해야 한다.13

| **지표 (Metric)** | **설명 및 모니터링 목적** |
| --- | --- |
| redisson_lock_acquisition_time | 락을 획득하기 위해 대기한 시간. 이 수치가 급증하면 시스템 병목이나 데드락 징조다. |
| redisson_lock_holding_time | 락을 점유하고 있는 시간. 이 시간이 길다면 트랜잭션 범위를 좁히거나 로직을 최적화해야 한다. |
| redis_connected_clients | Redis 연결 수. 스핀 락 구현 시 급격히 증가할 수 있으므로 감시가 필요하다. |
| lock_acquire_failures | 락 획득 실패 횟수. 비즈니스 로직의 정상 동작 여부를 판단하는 지표다. |

## **5. 장애 시나리오 시뮬레이션 및 대응 매뉴얼**

운영자는 발생 가능한 최악의 상황을 가정하고 시나리오별 대응책을 마련해야 한다.

### **5.1 시나리오 A: Redis 마스터 노드 크래시와 복제 지연**

- **상황**: Client A가 Master 노드에서 락을 획득했으나, 이 정보가 Slave로 복제되기 직전에 Master가 다운되었다. Sentinel이 Slave를 새로운 Master로 승격시켰으나, 락 정보가 없어 Client B가 동일한 리소스에 락을 획득한다.
- **영향**: 두 클라이언트가 동시에 임계 구역에 진입하여 데이터 오염 발생.
- **대응 및 복구**:
1. **사전 방어**: 중요한 데이터(결제 등)에는 반드시 **DB 유니크 제약조건**이나 **낙관적 락(Version check)**을 함께 사용하여 2중 잠금 장치를 마련한다.
2. **사후 복구**: 트랜잭션 로그를 분석하여 중복 처리된 건을 찾아 보상 트랜잭션(환불, 재고 복구)을 수행한다.
3. **아키텍처 개선**: 정합성이 매우 중요한 경우 Redlock을 도입하거나 Zookeeper로 마이그레이션한다.

### **5.2 시나리오 B: 클라이언트 GC Pause로 인한 데이터 오염**

- **상황**: Client A가 락을 잡고 로직을 수행하던 중 Full GC가 발생하여 10초간 멈췄다(Stop-the-world). 그 사이 Redis TTL(5초)이 만료되어 락이 해제되고, Client B가 락을 획득하여 데이터를 수정했다. GC에서 깨어난 Client A는 락이 풀린 줄 모르고 데이터를 덮어쓴다.
- **대응 (Fencing Token)**:
- **Fencing Token 패턴**: 락 서비스가 락을 발급할 때마다 단조 증가하는 토큰(버전 번호)을 함께 발급한다.
- Client A(Token 33)가 깨어나서 DB에 쓰기를 시도할 때, DB는 이미 Client B(Token 34)에 의해 업데이트되었음을 확인하고 Client A의 요청을 거부(Reject)해야 한다.2
- Redisson의 RFencedLock이 이 기능을 지원하므로 적극 활용한다.

### **5.3 시나리오 C: 락 타임아웃과 비즈니스 로직 불일치**

- **상황**: 외부 API 호출 지연으로 인해 로직 수행 시간이 락의 TTL을 초과했다.
- **대응**:
- **단기적**: Redisson Watchdog을 활성화하여 로직이 끝날 때까지 락을 자동 연장한다.
- **장기적**: 락 내부에서 외부 API 호출과 같은 불확실한 작업을 수행하지 않도록 설계를 변경한다. 락은 데이터 상태 변경과 같은 최소한의 범위로 축소하고, 외부 호출은 비동기로 처리하거나 락 밖으로 뺀다.

## **6. 결론**

Redis 분산 락은 마이크로서비스 아키텍처에서 데이터 정합성을 지키는 가장 강력하고 실용적인 도구다. 배달의민족, 토스, 쿠팡 등 선도 기업들의 사례는 Redis 락이 단순히 기술적 도구를 넘어, 선착순 이벤트나 실시간 결제와 같은 핵심 비즈니스 기능을 가능케 하는 인프라임을 증명한다.

성공적인 도입을 위해서는 **(1) Redisson과 같은 검증된 라이브러리를 사용하여 구현 복잡도를 낮추고**, **(2) Watchdog과 Fencing Token을 활용하여 GC 및 지연 상황에 대비하며**, **(3) DB 낙관적 락을 최종 방어선으로 구축하는 계층적 방어 전략**이 필수적이다. 또한, 그랩의 사례처럼 트랜잭션의 성격이 장기 실행(Long-running)으로 변모할 경우, 락에 집착하기보다 Temporal과 같은 워크플로우 엔진이나 Saga 패턴으로의 전환을 과감히 고려해야 한다.

결국 운영의 핵심은 '모든 것은 실패한다'는 가정 하에 Fallback과 회복 탄력성(Resiliency)을 설계에 내재화하는 것이다. 본 리포트가 제시한 사례와 전략들이 귀사의 분산 시스템을 더욱 견고하게 만드는 데 기여하기를 바란다.
