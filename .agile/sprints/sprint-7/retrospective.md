## 2. KPT (Keep, Problem, Try)

### Keep (좋았던 점)

*   **Best Fit 시나리오 설계의 성공:**
    Sprint 6의 "트럭과 스포츠카를 F1 서킷에서 경주시킨" 실수를 반복하지 않았습니다. 각 기술이 주인공이 되는 무대를 설계하니, 기존에 보이지 않던 인사이트가 쏟아졌습니다.

*   **리뷰를 통한 인사이트 심화:**
    테스트 결과를 기계적으로 기록하는 데 그치지 않고, 리뷰 과정에서 지속적으로 인사이트를 발굴했습니다.
    - "성공률 100%의 진짜 의미는 자원 낭비 제로" (시나리오 1 리뷰 중 발견)
    - "낙관적 락은 성능이 아니라 경합 감지 모니터링 도구" (시나리오 2 리뷰 중 발견)
    - "Lua Script 도입의 정당성은 TPS가 아니라 DB 커넥션 보호" (시나리오 4 리뷰 중 발견)

*   **테스트 격리 환경의 안정화:**
    `reset-infra`(Docker 완전 재시작)를 모든 시나리오에 적용하여 데이터 신뢰성을 확보했습니다. 시나리오 1, 2를 재측정했을 때 결과가 3% 이내 오차로 재현되어 방법론의 정확성을 검증했습니다.

### Problem (아쉬웠던 점)

*   **초기 시나리오 4 설계의 변경:**
    원래 "Redis as Primary Storage"로 계획했던 시나리오 4를 "Extreme Performance Comparison"으로 변경했습니다. Redis 단독 저장소 시나리오는 영속성 문제로 PoC 범위를 벗어났지만, 변경 과정에서 시간이 소요되었습니다.

*   **V2와 V3의 관계 정리 지연:**
    V3 리포트를 만들면서 V2를 업데이트할지 고민하는 시간이 있었습니다. 처음부터 "V2는 로그성 문서, V3가 최신"이라는 원칙을 세웠으면 더 깔끔했을 것입니다.

### Try (향후 개선 사항)

*   **시나리오 설계 시 변동 가능성 미리 고려:** 시나리오를 확정하기 전에 "이 시나리오가 PoC 범위 안에서 검증 가능한가?"를 체크리스트로 검증하면 설계 변경 비용을 줄일 수 있습니다.
*   **문서 버전 관리 원칙 선확립:** 리포트를 새로 만들 때 기존 문서와의 관계(대체/보완/로그)를 먼저 정의하면 혼란 없이 진행할 수 있습니다.

### 초기 기대 대비 예상 밖의 성과

*   **"통념 검증"이라는 차별점:** 단순 구현을 넘어 "낙관적 락이 비관적 락보다 빠르다"는 업계 통념을 데이터로 검증하고 반박한 것은 원래 계획에 없던 성과.
*   **점진적 최적화 전략 도출:** 0단계(@Version 모니터링) → 1단계(Pessimistic 타겟) → 2단계(Redis 스케일)이라는 실무 도입 프레임워크를 만들어낸 것.
*   **쓰기 최적화라는 상위 관점:** 동시성 제어를 쓰기 최적화(Write Optimization)의 한 축으로 재정의하고, 관리형 서비스 vs 자체 운영의 본질적 차이까지 통찰.

## 5. 애자일 & MVP 접근법 회고

### 왜 Sprint 단위로 나눴는가?

*   **복잡한 문제의 분해:** "동시성 제어 PoC"라는 큰 과제를 8개 Sprint로 쪼개니, 각 Sprint마다 명확한 목표와 Definition of Done이 생겼습니다. 한 번에 모든 것을 완성하려는 부담 없이 점진적으로 전진할 수 있었습니다.
*   **방향 수정의 유연성:** Sprint 6에서 "절대적 비교의 함정"을 발견하고 Sprint 7의 방향을 Best Fit 검증으로 전환한 것이 대표적입니다. 워터폴 방식이었다면 처음 계획에 얽매여 비효율적인 테스트를 반복했을 것입니다.

### MVP 마인드셋: 완벽보다 빠른 검증

*   Sprint 1-2에서 4가지 방식의 **최소 동작 구현**을 먼저 완성하고, Sprint 3-5에서 측정과 최적화를 반복했습니다.
*   처음부터 완벽한 성능 테스트를 설계하려 했다면, 구현도 못 끝냈을 것입니다. "일단 돌아가는 코드"가 있어야 "의미 있는 측정"이 가능했습니다.

### 선(先) 구현, 후(後) 연구의 학습 효과

*   Sprint 0-5에서 직접 삽질하며 겪은 경험(데드락, 커넥션 고갈, 재시도 폭주)이 Sprint 6의 실무 사례 연구를 **뼈에 사무치게** 만들었습니다.
*   "Redis Watchdog이 왜 필요한지"를 문서로 읽는 것과, 직접 타임아웃을 겪은 뒤 읽는 것은 이해의 깊이가 다릅니다.

### 작은 단위 반복(Iteration)의 가치

*   Sprint 7 내에서도 4개의 Iteration으로 세분화하여 진행했습니다. 시나리오 설계 → DB Lock 검증 → Redis 검증 → 문서화라는 흐름이 자연스럽게 이어졌습니다.
*   각 Iteration의 끝에서 리뷰를 진행하며 인사이트를 발굴하고 반영하는 **피드백 루프**가 문서의 품질을 높였습니다.

## 6. 핵심 기술 인사이트 (Lessons Learned)

### 기술적 인사이트

1. **"낙관적 락이 비관적 락보다 빠르다"는 조건부 명제.** 재시도가 거의 작동하지 않을 때만 미세하게 성립. 실무에서 실패를 허용하지 않는다면 비관적 락이 성능까지 앞섬.

2. **낙관적 락은 성능 도구가 아니라 경합 감지 모니터링 도구.** `@Version` 충돌 모니터링으로 비관적 락 적용 지점을 식별하는 것이 존재 이유.

3. **비관적 락의 성공률 100% = 자원 낭비 제로.** 모든 DB 커넥션이 비즈니스 가치를 생산. 낙관적 락의 재시도는 이전 커넥션 사용이 헛수고가 되는 자원 낭비.

4. **비즈니스 로직 지연이 기술 순위를 결정.** 100ms 지연 유무로 1위와 꼴찌가 완전히 뒤바뀜. 벤치마크가 아닌 비즈니스 맥락이 기술 선택의 기준.

5. **커넥션 풀 설정 하나가 아키텍처를 좌우.** DB Pool 50이면 Redis가 불필요하고, Pool 10이면 Redis가 필수. 서버를 늘리기 전에 `maximum-pool-size` 튜닝이 먼저.

6. **Redis 도입의 정당성: TPS가 아니라 커넥션 보호.** Lua Script를 위해 로직을 간소화하면 비관적 락도 26.7배 향상. Redis의 진짜 가치는 DB 자원을 근본적으로 보호하는 것.

7. **비관적 락은 Easy to Use, Hard to Master.** JPA `@Lock`은 간편하지만, MySQL Gap Lock, PostgreSQL Advisory Lock 등 RDBMS별 락 특성을 반드시 이해해야 함.

8. **동시성 제어는 쓰기 최적화의 출발점.** RDBMS 락 기반 전략 외에도 NoSQL, Redis 자료구조, CQRS/Event Sourcing 등 더 넓은 쓰기 최적화의 세계가 존재.

9. **병목 관리의 책임: 자체 운영 vs 관리형 서비스.** 커넥션 풀 기반 DB(RDBMS, MongoDB)는 엔지니어의 책임, HTTP API 기반 관리형 서비스(DynamoDB)는 비용으로 위임. 병목이 사라진 것이 아니라 누가 관리하느냐의 차이.

### 프로젝트 관리 인사이트

1. **선 구현 후 연구가 이해를 극대화한다.** 직접 삽질한 경험이 이론 학습의 배경지식이 되어 학습 효과를 배가시킴.

2. **절대적 비교가 아닌 상대적 비교가 인사이트를 준다.** "누가 제일 빠른가?"보다 "어떤 상황에서 누가 빛나는가?"가 실무적으로 의미 있는 질문.

3. **리뷰가 인사이트를 만든다.** 테스트 결과를 기록하는 것에서 끝나지 않고, 리뷰 과정에서 지속적으로 질문하고 토론하는 것이 핵심 인사이트를 발굴하는 원동력.

4. **테스트 격리는 선택이 아닌 필수.** `reset-infra` 없이 연속 테스트하면 이전 테스트의 잔재가 결과를 오염시킴. 측정의 신뢰성은 격리에서 시작.

5. **문서는 프로젝트 가치의 절반.** 아무리 좋은 코드와 테스트 결과도 문서화되지 않으면 공유할 수 없고, 공유할 수 없으면 가치가 반감.

## 7. 향후 확장 가능성

*   **쓰기 최적화 확장:** CQRS 패턴 도입, Event Sourcing을 통한 쓰기 분리, Redis Set/Sorted Set을 활용한 멱등성 보장 등 본 PoC의 인사이트를 기반으로 확장 가능.
*   **분산 환경 검증:** 단일 DB 인스턴스가 아닌 Read Replica, Sharding 환경에서의 동시성 제어 검증.
*   **모니터링 자동화:** `@Version` 충돌 모니터링을 Prometheus/Grafana로 자동화하여 실시간 경합 감지 대시보드 구축.
*   **관리형 서비스 비교:** DynamoDB Conditional Write vs RDBMS Lock의 실질적 비용/성능 비교.
