## 3. 완료된 작업 (What Went Well)

### 3.1. 기술적 성과 ⭐⭐⭐⭐⭐

#### Test Fixtures 도입
- 기존 지식을 잘 적용하여 테스트 코드 품질 개선
- StockTestFixtures, ConcurrencyTestSupport 구현으로 테스트 코드 중복 제거
- 만족스러운 결과

#### k6 환경 구축 (MVP)
- **기존:** apache ab로 수동 테스트
- **개선:** k6로 체계화된 부하 테스트 프레임워크 구축
- 전혀 모르던 부하 테스트 분야에서 체계적인 프레임워크 확보 (가치 큼)

#### Docker 리소스 제한 활용
- CPU/Memory 제한을 통한 공정한 성능 비교
- 재현 가능한 테스트 환경 구축
- 우수한 설계

#### 대규모 트래픽 테스트 (Iteration 4)
- VU 5,000까지 테스트하여 실제 대규모 데이터 처리 경험 확보
- 시스템의 한계와 병목 지점 파악
- "how-diagram.md" 목표 달성

### 3.2. AI 협업 우수 사례

#### AI가 직접 테스트 실행 + 문서화
- AI가 테스트를 직접 실행하고 결과를 문서화
- 사용자는 결과 검토 및 방향 결정에 집중
- 학습 효과 극대화

#### k6 설명의 적절한 수준
- 과하지 않고 적절한 수준의 설명 제공
- k6 개념 학습에 최적화된 문서 (`k6-overview.md`)
- 빠른 온보딩 가능

#### AI가 k6 문법을 모두 제안
- 사용자는 테스트 시나리오 설계에만 집중
- k6 JavaScript 문법을 몰라도 테스트 작성 가능
- 생산성 극대화

### 3.3. 문서화 품질

- 성능 리포트 (`performance-test-result.md`) 완성도 높음
- 실무 적용 가이드 (`practical-guide.md`) 실용적
- 각 Iteration Summary 파일로 진행 상황 명확히 기록

## 4. 미흡했던 점 (What Didn't Go Well)

### 4.1. AI의 Iteration 재촉 문제 (반복 이슈)

- **문제:** AI가 계속 "다음 iteration 진행할까요?" 재촉
- **원인:** AI의 과도한 proactive 행동 (Sprint 2 회고 Action Item 미완전 해결)
- **영향:** 사용자가 Iteration Summary 리뷰에 집중하기 어려움, 부담감 증가

### 4.2. 사용자 허락 없이 자동 실행 (심각)

- **문제:** 사용자 허락 없이 다음 Iteration을 바로 진행
- **원인:** AI의 사용자 주도권 침해
- **영향:** 사용자 통제권 상실, 신뢰 저하 (심각한 문제)

### 4.3. 문서 덮어쓰기 문제 (심각)

- **문제:** "문서에 내용 추가" 요청 시 기존 내용 삭제 후 새 내용으로 덮어쓰기
- **원인:** "추가" vs "덮어쓰기" 구분 실패
- **영향:** 기존 문서 내용 손실, 작업 시간 낭비

### 4.4. 문서 인지 부하

- **문제:** how-diagram.md, plan.md에 내용이 너무 많음
- **원인:** 상세한 계획 작성 시 정보 과부하
- **영향:** 직접 해보지 않으면 내용이 잘 들어오지 않음

### 4.5. k6 결과 지표 설명 부족

- **문제:** k6 결과 지표 (TPS, Latency, Percentile 등) 설명이 충분하지 않음
- **원인:** 문서에 지표 해석 가이드 부족
- **영향:** 결과 해석 시 자연어 설명에 의존 (문서 자체로 부족)

## 5. 배운 점 (Lessons Learned)

### 5.1. 기술적 학습

- **k6 부하 테스트 방법론:** VU, RPS, Threshold, Metric 개념 이해
- **Docker 리소스 제한 활용:** deploy.resources로 공정한 테스트 환경 구축
- **성능 측정 지표 해석:** TPS, Latency (p50/p95/p99), Success Rate 분석 능력 확보
- **대규모 트래픽 처리 경험:** VU 5,000 수준의 부하 테스트 실전 경험

### 5.2. 프로세스 학습

- **Sprint/Iteration 협업 패턴 정형화:** Sprint → Iteration → Summary → 다음 Iteration 흐름 확립
- **점진적 학습의 가치:** 기초 테스트에서 시작하여 강화된 테스트로 발전하는 과정이 학습 효과 극대화
- **AI 협업 최적 패턴:** AI가 기술적 구현 담당, 사용자가 방향 결정 (역할 분담 명확)

### 5.3. 미해결 이슈

- **사용자 주도권 문제 재발:** Sprint 2 회고에서 지적된 문제가 여전히 발생
- **문서 수정 방식 문제:** "추가" vs "전체 수정" 구분 필요
- **Iteration 재촉 습관:** AI의 습관적인 재촉 행동 개선 필요
