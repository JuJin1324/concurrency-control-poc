## 3. 완료된 작업 (What Went Well)

### 3.1. 프로젝트 정합성 검증의 가치

Sprint 0에서 작성한 brainstorm.md와 실제 Sprint 0~3 수행 내용을 비교하여 **기획 대비 달성률을 명확히 파악**했습니다.

**성과:**
- 핵심 KPI 달성 여부 확인 (4가지 동시성 제어 방법 구현 완료, 성능 테스트 데이터 확보)
- 누락 항목 파악 (Grafana 대시보드, CI/CD 파이프라인 등은 의도적 제외)
- 프로젝트 범위의 명확화 → 완성도에 대한 확신

**배운 점:**
- 주기적인 정합성 검증은 프로젝트 방향성을 유지하는 데 필수
- "무엇을 하지 않을 것인가"를 명확히 하는 것도 중요

### 3.2. Hell Test를 통한 극한 상황 검증

"재고 100개 vs 유저 5,000명" 시나리오로 선착순 이벤트 상황을 재현하여, 각 방법의 **극한 상황에서의 거동**을 확인했습니다.

**성과:**
- 4가지 방법(Lua, Optimistic, Pessimistic, Redis Lock)의 Contention 상황 비교 데이터 확보
- 각 방법의 한계점 및 적합한 사용 케이스 명확화
- Hell Test 가이드 문서화 완료 (`docs/test-guides/hell-test.md`)

**배운 점:**
- 일반 부하 테스트만으로는 동시성 제어의 극한 성능을 파악하기 어려움
- Hell Test는 "최악의 시나리오"에서의 안정성 검증에 필수

### 3.3. README.md 전면 개편

기존의 기술 중심 README를 **사용자 중심 온보딩 문서**로 전환하여, 프로젝트의 첫인상을 크게 개선했습니다.

**성과:**

- 불필요한 상세 리포트 링크 정리 → 인지 부하 감소

**배운 점:**
- README는 "코드 설명서"가 아니라 "프로젝트 첫인상"
- "무엇을 할 수 있는가"를 먼저 보여주고, "어떻게 하는가"는 나중에

## 5. 배운 점 (Lessons Learned)

### 5.1. 문서화는 프로젝트 완성도의 50%

코드만 완성되어도 프로젝트는 50%에 불과합니다. README, 블로그 포스팅, 아키텍처 문서 등의 **문서화**가 프로젝트의 가치를 완성시킵니다.

### 5.2. 정합성 검증의 중요성

프로젝트 중간중간 "기획 대비 실제 수행"을 비교하는 **정합성 검증**은 방향성을 유지하는 데 필수입니다.

### 5.3. Hell Test는 동시성 제어의 필수 검증

일반 부하 테스트만으로는 동시성 제어의 극한 성능을 파악하기 어렵습니다. **Hell Test**를 통해 최악의 시나리오를 검증해야 합니다.

### 5.4. README는 첫인상

README는 "코드 설명서"가 아니라 "프로젝트 첫인상"입니다. **Quick Start**를 최상단에 배치하고, 사용자 중심으로 작성해야 합니다.

### 5.5. 블로그 포스팅 작성은 프로젝트 이해도를 높임

프로젝트를 외부에 설명하기 위해 블로그 포스팅을 작성하는 과정에서, 프로젝트의 핵심 인사이트를 더욱 명확히 정리할 수 있었습니다.

## 10. 다음 Sprint 준비사항

### Sprint 5 목표 (how-diagram.md 기준)

**Sprint 5:** 심화 연구 (실무 도입 사례) - **개발 중심 → 운영 중심 전환**

**목표:** 각 동시성 제어 방식의 실제 현업 도입 사례 연구 및 분석

**Sprint 0-4 vs Sprint 5 관점 차이:**
- **Sprint 0-4 (개발 중심):** "어떻게 구현하는가?" - 코드 작성, 테스트, 성능 측정
- **Sprint 5 (운영 중심):** "어떻게 운영하는가?" - 실무 사례, 장애 대응, 모니터링, 트레이드오프

**내용 (운영 관점):**
- **Pessimistic Lock:**
  - 금융권 계좌 이체, 재고 관리 시스템 등 정합성이 극도로 중요한 사례
  - 운영 시 고려사항: Deadlock 감지/해결, Lock Timeout 설정, 모니터링

- **Optimistic Lock:**
  - 위키 편집, 자원 경합이 낮은 일반 웹 서비스 사례
  - 운영 시 고려사항: 충돌 발생률 모니터링, Retry 전략, 사용자 경험

- **Redis Lock:**
  - 분산 환경에서의 분산 락 적용 사례 (e.g. 배민 선착순 쿠폰, 토스 주문 결제)
  - 운영 시 고려사항: Redis 장애 시 대응, Lock 누수 방지, TTL 설정

- **Lua Script:**
  - 초고부하 선착순 이벤트 처리 사례 (e.g. 쿠폰 발급, 티켓팅)
  - 운영 시 고려사항: Script 버전 관리, 성능 모니터링, Fallback 전략

**산출물:**
- 기술 블로그 심화편 (Case Study + 운영 경험)
- 방식별 대표 사용 사례 정리 리포트 (운영 관점 포함)
- 각 방식의 장애 시나리오 및 대응 방안 정리

**Sprint 5의 가치:**
- 단순 PoC를 넘어 "실무 적용 가능성"을 보여주는 깊이 있는 분석
- 면접 시 "왜 이 방식을 선택했는가?"에 대한 실무 사례 기반 답변 가능
- 포트폴리오를 "학습용 토이 프로젝트"에서 "실무 운영 가능한 케이스 스터디"로 승격
- **"개발만 하는 개발자"가 아닌 "운영까지 고려하는 시니어 개발자" 어필**
